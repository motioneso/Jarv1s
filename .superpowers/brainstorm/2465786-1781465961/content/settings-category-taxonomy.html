Proposed Settings Taxonomy

Split the experience into Personal Settings and Admin / Setup. The designer can choose layout, but these are the product categories we should brief them on.

Personal Settings
Profile & account
Email, display name, role, sign-out/security affordances.
Connected accounts
Google and future sources. Status, scopes, reconnect, revoke.
Assistant & AI
Shared assistant vs personal provider, configured models, capability defaults.
Memory & context
Recall, remembered facts, user corrections, what Jarvis believes.
Modules & data sources
Turn optional modules on/off for self; understand what each module contributes.
Workflow preferences
Task default view now; later notification sensitivity, briefing preferences, tone/persona.
Admin / Setup
People & access
Users, pending approvals, active/deactivated state, admin promotion/demotion.
Registration & identity
Open registration, approval requirement, login provider availability.
Instance modules
Enable/disable optional modules across the instance; required modules are locked.
Connector admin
Safe metadata across connector accounts, not private synced data or secrets.
Host setup
Terminal multiplexer and other local runtime checks. Likely setup/advanced, not daily settings.
Audit & operations
Admin audit trail, exports/deletion references, backup/restore references.

Design Note

Avoid exposing backend nouns as top-level categories. Users should see control areas: connections, assistant behavior, memory/context, modules, notifications, and access. Provider/model/manifest/multiplexer details can exist inside advanced rows or setup flows.
