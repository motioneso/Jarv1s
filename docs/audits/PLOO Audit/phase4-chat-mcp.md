#### [MED] Claude CLI launch omits the spike-verified native-tool disable (`--tools ""` / `--disallowedTools`) on the production MCP path
**File:** packages/chat/src/live/cli-chat-engine.ts:207
**Invariant violated / concern:** Phase 2 security invariant (spec §7): "Native shell, file, edit, and web tools are OFF" — native-tool lockdown must be harness-enforced (not model-/prompt-injection-bypassable), per the MCP-client-support spike's verified Claude recipe.
**Detail:** `buildClaudeCommand` has two branches. When MCP is configured (the production path — packages/module-registry/src/index.ts:149 always passes `mcpServerUrl`), it emits ONLY `--permission-mode default`, `--mcp-config <json>`, `--allowedTools "mcp__jarvis__*"`, `--append-system-prompt-file`, `--session-id`, `--strict-mcp-config`. It never emits `--tools ""` or `--disallowedTools`; `--tools ""` lives only in the NON-MCP else branch (line 224). The verified spike recipe (docs/superpowers/spikes/2026-06-08-mcp-client-support.md:38) is `--allowedTools "mcp__jarvis__*"` + deny natives (`--disallowedTools "Bash" "Edit" ...` or `--tools ""`), and stresses that the deny rules are "harness-enforced, NOT model-enforced -> not prompt-injection-bypassable." `--allowedTools` alone is an auto-APPROVE list, not a hard disable; `--strict-mcp-config` only restricts MCP server sources, not native tools. So on the production path native tools (Bash, Read, WebFetch/WebSearch, Edit/Write) are NOT harness-disabled — they fall back to `--permission-mode default` prompting inside a programmatically-driven tmux pane with no human at the TUI, a posture the spike did not verify and which contradicts both the spec invariant and the spike recipe. Since the chat agent ingests untrusted/injected content (recall memories, prior turns, tool outputs), this weakens the single chokepoint's prompt-injection mitigation. The file header comment (lines 16-17) still claims `--tools ""` disables all native tools, which is stale/misleading for the MCP branch.
**Verification:** Downgraded HIGH→MED. Skeptic independently confirmed the MCP branch (lines 210-222) emits `--permission-mode default` + `--mcp-config` + `--allowedTools "mcp__jarvis__*"` and never `--tools ""`/`--disallowedTools`; production always takes the MCP branch (module-registry index.ts:149 hardwires `mcpServerUrl`, chat-session-manager.ts:153-158 mints token + passes both); the stale header comment is real; and the omission is even pinned by tests/unit/cli-chat-engine.test.ts:31 asserting the MCP branch does NOT contain `--tools ""`. Counter-evidence on severity: spec §7 (docs/superpowers/specs/2026-06-08-jarvis-chat-phase2-agentic-mcp.md:160-162) deliberately replaces the bypassable `--tools ""` denylist with an allowlist; the launch uses `--permission-mode default` (NOT bypass), so an injected native-tool request would surface an approval prompt in a headless pane with no human to approve — it blocks rather than executes. Rationale: a real defense-in-depth gap (allowlist present, the spike's belt-and-suspenders hard-deny absent) but no demonstrated path to native-tool execution → MED, not HIGH.
**Suggested fix:** In the MCP branch, additionally push the harness-enforced native-tool disable exactly as the spike prescribes — either `--tools ""` or `--disallowedTools "Bash" "Edit" "Read" "Write" "WebFetch" "WebSearch" "Glob" "Grep"` (plus `Task`) — alongside `--allowedTools "mcp__jarvis__*"`. Add an integration assertion that the assembled Claude launch line contains a deny/empty-allowlist token whenever MCP is wired, and fix the header comment.

#### [MED] Chat episodic-recall embed job runs as `jarvis_worker_runtime`, but `memory_chunks`/`memory_file_index` RLS policies target only `jarvis_app_runtime` — worker has table grants but no applicable policy, so reads/writes fail closed and chat history is never embedded
**File:** packages/memory/sql/0040_memory_chat_source.sql:15
**Invariant violated / concern:** Private by default / RLS-correctness: every runtime role that must touch a FORCE-RLS table needs both a GRANT and an applicable RLS policy; a grant without a matching policy yields deny-all under PostgreSQL.
**Detail:** The chat embed-turn job (packages/chat/src/jobs.ts:41-94, registered via `registerChatJobWorkers` at jobs.ts:120) runs in the worker process, which connects as `jarvis_worker_runtime` (packages/db/src/urls.ts:27-29). `handleEmbedTurnJob` calls `memoryRepository.getFileIndex` (FROM app.memory_file_index, repository.ts:134), `upsertFileChunks` (INSERT INTO app.memory_chunks, repository.ts:38) and `upsertFileIndex` (INSERT INTO app.memory_file_index, repository.ts:154). Migration 0040 (lines 15-16) GRANTs SELECT/INSERT/UPDATE/DELETE on both tables to `jarvis_worker_runtime`, but the RLS policies on `memory_chunks` (0030_memory_index.sql:44-63) and `memory_file_index` (0032_memory_embedding_768.sql:42-61) are all written `FOR ... TO jarvis_app_runtime` only. With FORCE ROW LEVEL SECURITY enabled (0030:40, 0032:40) and no permissive policy applicable to the worker role, PostgreSQL denies all rows on SELECT and rejects every INSERT/UPDATE for `jarvis_worker_runtime`. Net effect: `getFileIndex` returns nothing and the upserts are rejected, so chat turns are never embedded into `memory_chunks(source_kind='chat')` — and recall (`RecallService.recallEpisodic`, recall-port.ts:68-112) therefore never retrieves any chat history. No later migration repairs this (only 0043/0044 connectors and 0045/0046 auth follow). Note the inconsistency that masks the bug: the sibling `chat_memory_facts` policies (0041:28-42) were written with NO role target (apply to PUBLIC) so the worker can use them, while the `memory_chunks`/`file_index` policies were scoped `TO jarvis_app_runtime`. This is fail-closed (no cross-user leak), so it is a foundational correctness/RLS-role defect defeating the Phase 4 recall feature rather than a confidentiality breach.
**Verification:** Confirmed MED. Skeptic verified every link: the worker grant without a worker policy (0040:15-16), `memory_chunks` FORCE RLS with all four policies `TO jarvis_app_runtime` (0030:40,45-63), `memory_file_index` likewise (0032:40,43-61), the role-agnostic PUBLIC `chat_memory_facts` policies that mask the inconsistency (0041:29-45), the worker connecting as `jarvis_worker_runtime` (worker.ts:12-19, urls.ts:27-29), no `SET ROLE` anywhere in the data-context/pool path (data-context.ts:30-38 only `set_config` of `app.actor_user_id`/`app.request_id`; database.ts:14-26 plain pool), the embed handler being live-wired (module-registry index.ts:152-153 → jobs.ts:120-143 → handler at 61-93), and the downstream empty-recall impact (recall-port.ts:75-79). Under PostgreSQL, FORCE RLS + table GRANT + no applicable policy = deny-all (SELECT zero rows, INSERT/UPDATE rejected on WITH CHECK). Rationale: a genuine RLS grant/policy-pairing invariant violation that silently disables Phase 4 episodic recall; fail-closed (no cross-user leak, no secret escape) so below CRIT/HIGH, but well above LOW hygiene → MED.
**Suggested fix:** Add `jarvis_worker_runtime` to the role list of the `memory_chunks` and `memory_file_index` SELECT/INSERT/UPDATE policies in a NEW migration (mirror the chat 0036 pattern: `TO jarvis_app_runtime, jarvis_worker_runtime`, preserving the `owner_user_id = app.current_actor_user_id()` USING/WITH CHECK expressions verbatim). Do not edit the applied 0030/0032/0040 files. Add an integration assertion that an embed-turn job executed as the worker actually inserts a `memory_chunks` row.

#### [LOW] `chat_user_memory_settings` RLS policies omit the role target and the `IS NOT NULL` actor guard used by every other chat policy
**File:** packages/chat/sql/0042_chat_memory_settings.sql:16
**Invariant violated / concern:** Private-by-default RLS consistency: policies should be scoped to the runtime role(s) and guarded so an unset actor matches no rows, matching the rest of the module (0014/0025/0036).
**Detail:** The four policies on `app.chat_user_memory_settings` (select/insert/update/delete) are declared without `TO jarvis_app_runtime` (so they default to PUBLIC) and without the `app.current_actor_user_id() IS NOT NULL AND ...` guard that 0014/0025/0036 use. In practice this is not exploitable today: `app.current_actor_user_id()` returns NULL when the GUC is unset (infra/postgres/migrations/0002_app_rls.sql:11-13), so `user_id = NULL` yields NULL (no rows), and table privileges are GRANTed only to `jarvis_app_runtime`; FORCE RLS also applies the predicate to the table owner. So rows stay owner-scoped. It is a consistency/hardening gap, not a live leak.
**Verification:** Confirmed LOW. Skeptic opened 0042 and the three comparison files: 0042:16-29 are all bare (no `TO jarvis_app_runtime`, no `IS NOT NULL` guard), while 0014:178-185, 0025:11-21, and 0036:21-31 all carry both. Non-exploitability corroborated four ways: (1) `current_actor_user_id()` returns NULL when unset (0002:11-13) → equality yields zero rows; (2) FORCE RLS covers the owner (0042:13); (3) GRANTs go only to `jarvis_app_runtime` (0042:31), and `jarvis_worker_runtime` is never granted on this table, so the PUBLIC target is unreachable by any other role; (4) the repository accepts only `DataContextDb` (memory-settings-repository.ts:3,24,42). Rationale: a defense-in-depth divergence from the module's pattern with no live leak and no plausible exploit path → LOW.
**Suggested fix:** In a NEW migration (never edit the applied 0042), recreate the four policies as `... TO jarvis_app_runtime USING (app.current_actor_user_id() IS NOT NULL AND user_id = app.current_actor_user_id())` to match the rest of the module's pattern.

#### [LOW] `chat_messages_update` RLS policy omits the `app.current_actor_user_id() IS NOT NULL` guard that the SELECT/INSERT policies carry — USING/WITH CHECK consistency gap
**File:** packages/chat/sql/0036_chat_worker_runtime_grants.sql:54
**Invariant violated / concern:** Defense-in-depth consistency: a NULL actor must never satisfy an owner predicate; every owner-scoped policy on a table should guard actor `IS NOT NULL` identically so a missing-actor context cannot interact with the `= NULL` SQL three-valued-logic edge.
**Detail:** In 0036 the `chat_messages_update` policy is `USING (owner_user_id = app.current_actor_user_id()) WITH CHECK (owner_user_id = app.current_actor_user_id())` (lines 58-59), whereas `chat_messages_select` (0036:39-49) and `chat_messages_insert` (0014:218-231) both prefix `app.current_actor_user_id() IS NOT NULL AND ...`. If the actor GUC is unset, `owner_user_id = current_actor_user_id()` evaluates to NULL → treated as false, so the policy fails closed today (not exploitable). But the asymmetry is a latent footgun: any future change that makes `current_actor_user_id()` return a sentinel or that combines this with another permissive predicate could silently widen UPDATE scope. The same missing-guard pattern recurs on `chat_user_memory_settings` (0042) and `chat_memory_facts` (0041) policies — and those additionally lack a `TO` role target — which is the very inconsistency that produced the worker deny-all in the MED finding above.
**Verification:** Confirmed LOW. Skeptic verified 0036:54-59 has no `IS NOT NULL` prefix while same-table siblings do (select 0036:40-41; insert 0014:222-231), and that the original UPDATE policy (0035_chat_messages_update_grant.sql:12-13) was always missing it too (0036 only widened the role list). Crucially, `current_actor_user_id()` returns a true SQL NULL (not a sentinel UUID) when the GUC is unset (0002:11-12), so `owner_user_id = NULL` can never be true under three-valued logic — the policy fails closed identically to the guarded version, no runtime exposure path today. Cross-references partially accurate: 0042:17,21,25,29 and memory/sql/0041_memory_facts.sql:30,34,38,42 likewise omit the guard and the `TO` clause; the finding mislabels 0041 as a chat-module file (it lives in packages/memory/sql/), but the substantive claim is correct. Rationale: hardening/hygiene asymmetry with NULL semantics positively ruling out an exploit → LOW, not MED.
**Suggested fix:** In a new migration, recreate `chat_messages_update` with the `app.current_actor_user_id() IS NOT NULL AND` prefix on both USING and WITH CHECK, matching the select/insert policies; while there, normalize the `chat_user_memory_settings` (0042) and `chat_memory_facts` (0041) policies to include an explicit `TO jarvis_app_runtime` role target and the `IS NOT NULL` guard for table-wide consistency. Never edit the applied 0036/0041/0042 files.
