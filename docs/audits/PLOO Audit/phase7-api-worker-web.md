#### [HIGH] Hardcoded development BETTER_AUTH secret signs sessions whenever NODE_ENV is not exactly "production"
**File:** packages/auth/src/index.ts:354
**Invariant violated / concern:** Secrets never escape / session tokens must be unforgeable; auth signing keys must be high-entropy and never shipped in source.
**Detail:** `readAuthSecret()` returns the literal string `"jarv1s-development-better-auth-secret"` (line 365) whenever neither `BETTER_AUTH_SECRET` nor `AUTH_SECRET` is set AND `env.NODE_ENV !== "production"` (line 361). better-auth uses this secret to sign/verify session cookies and tokens. Because the guard keys strictly on the string `"production"`, any deployment that leaves `NODE_ENV` unset, blank, or set to `"prod"`/`"development"`/`"staging"` silently runs with a publicly-known signing key committed to the repository. Jarv1s is explicitly a LAN self-host product (per dev-environment notes and the `--host` LAN guidance) where operators routinely start the API without a tuned `NODE_ENV`; anyone who can reach the LAN port and reads this constant can forge a valid session cookie for any user, fully defeating authentication and RLS scoping (which trusts the resolved `actorUserId`). The blast radius is total account takeover. The first auditor flagged the rate-limit XFF fallback and Vite Origin rewrite but did not cover the auth signing key.
**Verification:** Confirmed. packages/auth/src/index.ts:354-366 — `readAuthSecret()`: `const secret = env.BETTER_AUTH_SECRET ?? env.AUTH_SECRET; if (secret) return secret; if (env.NODE_ENV === "production") throw ...; return "jarv1s-development-better-auth-secret";`. The throw guard at line 361 fires ONLY when `NODE_ENV` is exactly `"production"`; the hardcoded literal is at line 365. packages/auth/src/index.ts:134 — `secret: readAuthSecret(env)` feeds this directly into `createBetterAuthOptions`, i.e. better-auth's master session/token signing key. The identical `env.NODE_ENV === "production"` gate at packages/db/src/keyring.ts:29 confirms this is the codebase's intended production discriminator with no stricter upstream check; `infra/env.production.example` sets both `NODE_ENV=production` and a real `BETTER_AUTH_SECRET`. No startup assertion anywhere rejects the known dev literal outside production. Rationale: the fallback returns a publicly-committed signing key whenever both secret env vars are unset and `NODE_ENV` is anything other than the exact string `"production"`; that key signs/verifies sessions, so a reachable attacker who reads this repo constant can forge a session for any user — total account takeover. Correctly NOT a CRIT (latent, not exploitable on a correctly-configured deployment where production sets a real secret), but a HIGH given the headless LAN self-host model where `NODE_ENV` is trivially left unset, the zero defense-in-depth, and the complete blast radius.
**Suggested fix:** Require an explicitly-provided secret in all environments (throw when unset regardless of `NODE_ENV`), or at minimum (a) generate a random per-process secret on the fly when unset in dev so it is never a known constant, and (b) loudly warn that sessions will not survive restart. Never embed a constant signing key in source.

#### [MED] Admin/identity tables have no RLS — the "RLS applies to all actors" invariant is not honored for them
**File:** infra/postgres/migrations/0004_auth_workspaces_settings.sql:86
**Invariant violated / concern:** No admin private-data bypass — RLS applies to ALL actors including admins; private by default with a DB-level backstop, not just application conventions.
**Detail:** Migration 0004 grants SELECT/INSERT/UPDATE on `app.workspaces`, `app.workspace_memberships`, `app.resource_grants`, `app.instance_settings` to `jarvis_app_runtime` (lines 86-88) and 0005 creates `app.admin_audit_events`, but NO migration anywhere (0001-0046) ever runs `ENABLE ROW LEVEL SECURITY` or defines a policy on these tables. Grepping all of `infra/postgres/migrations/*.sql` for `ROW LEVEL SECURITY`/`POLICY` against these table names yields zero hits; the only RLS-enabled app tables are `rls_probe_items` (0002), `users`/`auth_accounts`/`better_auth_sessions` (0045), `auth_sessions`/`auth_verifications` (0046). This means `jarvis_app_runtime` has unconditional read/write on every user's workspace memberships, resource grants, and the full admin audit log with no DB-level scoping. The ONLY thing preventing cross-user exposure is the application-layer `requireAdmin()` check (packages/settings/src/routes.ts:331) plus per-query `where user_id =` filters (e.g. `listMembershipsForUser`, repository.ts:95). A single missing `requireAdmin` or a query that forgets its owner filter exposes all users' data with no backstop — exactly the failure mode the RLS invariant exists to prevent. Contrast the connectors module which DOES scope its admin read via `withDataContext` + a SECURITY DEFINER safe-metadata function (routes.ts:217).
**Verification:** Confirmed at MED. GRANTs verified at infra/postgres/migrations/0004_auth_workspaces_settings.sql:86-88 and 0005_admin_audit_events.sql:18,20. No RLS anywhere on these 5 tables — grep returns zero `ENABLE ROW LEVEL SECURITY`/`POLICY` hits against them; the only RLS-enabled app tables are `rls_probe_items` (0002), `shares` (0017), `users`/`auth_accounts`/`better_auth_sessions` (0045), `auth_sessions`/`auth_verifications` (0046). 0028 (Slice 1f teardown) dropped `workspace_id` columns and membership functions but did NOT drop the tables or revoke the grants. Only protection is app-layer: `SettingsRepository` (packages/settings/src/repository.ts:64) takes a raw `Kysely<JarvisDatabase>` handle and runs unscoped reads with no `withDataContext`/`SET LOCAL` actor context; admin routes are gated only by `requireAdmin()` (routes.ts:331-344), and the actor-scoped filter at repository.ts:95/99 is the only thing protecting the unguarded `/api/me` route. `appDb` connects as `jarvis_app_runtime` (packages/db/src/urls.ts:21-23, wired in apps/api/src/server.ts:32-37,99). Rationale: a genuine defense-in-depth gap — the "RLS applies to all actors / DB-level backstop" invariant is not honored for these tables — but MED is correct: no exploitable path today (every admin list route carries a working `requireAdmin()`, the only unguarded route `/api/me` is hard-scoped to the actor's own rows, no secrets live in these tables), a textbook missing-by-default protection not proven exploitable.
**Suggested fix:** Enable + FORCE RLS on `workspaces`, `workspace_memberships`, `resource_grants`, `instance_settings`, and `admin_audit_events` with explicit per-role policies (e.g. self-row `USING (user_id = app.current_actor_user_id())` for member-scoped reads, and an admin-scoped policy gated on a trusted `is_instance_admin` signal rather than relying solely on the app-layer `requireAdmin`). This makes "private by default" structural rather than conventional, matching the pattern already used for users/auth tables in 0045/0046.

#### [MED] SettingsRepository accepts a raw Kysely handle and runs admin/identity queries outside any DataContext (actor GUC never set)
**File:** packages/settings/src/repository.ts:64
**Invariant violated / concern:** DataContextDb only — every repository accepts only a branded `DataContextDb` handle, never a raw Kysely instance; all scoped reads run inside `withDataContext` so the actor GUC is set.
**Detail:** `SettingsRepository`'s constructor is `constructor(private readonly db: Kysely<JarvisDatabase>)` (repository.ts:64) and `registerSettingsRoutes` wires it with the root `appDb`: `new SettingsRepository(dependencies.appDb)` (routes.ts:75). Every method (`getUserById`, `listUsers`, `listWorkspaces`, `listMembershipsForUser`, `createWorkspace`, `upsertWorkspaceMembership`, `upsertResourceGrant`, `setInstanceSetting`, `insertAuditEvent`, etc.) executes directly on this raw handle, NEVER inside `DataContextRunner.withDataContext`, so `app.actor_user_id` is never set for these queries. This is the structural twin of the no-RLS finding above: because the actor GUC is absent, even if RLS were enabled on these tables, `app.current_actor_user_id()` would return NULL and policies keyed on it would deny everything — so the module is architecturally committed to running unscoped. Cross-user safety rests entirely on the application-layer `requireAdmin()` check and hand-written `where user_id = $actor` filters. It works today only because `users` has a deliberate `USING(true)` SELECT policy (0045:83) and the other tables have no RLS at all.
**Verification:** Confirmed at MED. packages/settings/src/repository.ts:64 takes a raw root Kysely handle, not a branded `DataContextDb`; every method runs `this.db.selectFrom(...)`/insert/transaction directly (e.g. getUserById :73-75, listUsers :77-84, listWorkspaces :86-93, listResourceGrants :262-271, createWorkspace :138, upsertResourceGrant :273, insertAuditEvent :481-505), none calling `withDataContext`. routes.ts:75 wires `new SettingsRepository(dependencies.appDb)` and apps/api/src/server.ts:32-37 creates `appDb` via `createDatabase({...})` (the root pool). packages/db/src/data-context.ts:30-38 — `withDataContext` is the ONLY code that runs `set_config('app.actor_user_id', ...)` (setLocal :62-68) inside a transaction and returns the branded handle; bypassing it means the GUC is never set. RLS confirmed absent on all five settings tables; `app.users` gets a deliberate permissive SELECT policy `USING (true)` at 0045:83-87. No `BYPASSRLS` on runtime roles (bootstrap/0000_roles.sql uses `NOBYPASSRLS`). Rationale: a real DataContextDb-invariant violation and defense-in-depth gap (the DB has no fallback if a `requireAdmin` gate or owner filter were dropped), but not exploitable today — every admin route is gated by `requireAdmin` (re-reads `is_instance_admin` per request) and the only non-admin path `/api/me` scopes all three repository calls to the caller's own `accessContext.actorUserId` (routes.ts:86-98); no secret columns live on these tables. Squarely MED.
**Suggested fix:** Route the actor-scoped settings reads/writes (`/api/me`, `listMembershipsForUser`, `listWorkspacesForUser`, `createWorkspace`, membership/grant mutations) through `DataContextRunner.withDataContext` so the actor GUC is set and RLS can enforce ownership. Keep genuinely cross-user admin reads (`listUsers`/`listWorkspaces`) behind `requireAdmin` AND a DB policy that checks an admin signal, rather than a raw root-Kysely handle that bypasses the data-context layer entirely.

#### [LOW] Unauthenticated /api/bootstrap/status discloses exact total user count of the instance
**File:** packages/settings/src/routes.ts:77
**Invariant violated / concern:** Private by default — no unauthenticated route should leak instance-internal state beyond the minimum needed for the bootstrap UX.
**Detail:** `GET /api/bootstrap/status` (lines 77-84) has no `resolveAccessContext` / auth guard and returns `{ needsBootstrap, userCount }` where `userCount` is the exact total number of registered users on the instance (`repository.countUsers()`). The bootstrap UI only needs the boolean `needsBootstrap` (`userCount === 0`). Returning the precise count to any unauthenticated caller leaks instance population — minor on a single-operator LAN box, but it is gratuitous internal-state disclosure on an unauthenticated endpoint, the same class of concern the audit's health-endpoint question targets. The first auditor's findings did not cover this endpoint's payload.
**Verification:** Confirmed at LOW. packages/settings/src/routes.ts:77-84 — the handler takes no `(request, reply)` args and never calls `dependencies.resolveAccessContext`, unlike `/api/me` (routes.ts:88) and every `/api/admin/*` route (via `requireAdmin`, routes.ts:336). The response schema (packages/shared/src/platform-api.ts:367-379) returns `userCount: { type: "number" }`. `repository.countUsers()` (repository.ts:66-71) runs `SELECT app.count_all_users()`, defined SECURITY DEFINER in 0045_auth_secret_rls.sql:127-135 as `SELECT count(*) FROM users` — the exact total. No global auth hook backstops it (apps/api/src/server.ts registers routes inside `server.after()` with no `onRequest`/`preHandler` auth hook, server.ts:67-107). Rationale: real gratuitous internal-state disclosure on an unauthenticated route, but correctly LOW — breaks no hard invariant, exposes no private user data/secret/per-user record (the count is an aggregate), grants no access; on the single-operator-LAN model the leaked figure is a small household headcount to a caller who can already reach the LAN endpoint. A trivially-fixed hygiene gap.
**Suggested fix:** Return only the boolean the bootstrap flow requires: `{ needsBootstrap: userCount === 0 }`. Drop the raw `userCount` from the unauthenticated response (or gate the count behind admin auth if a count is genuinely needed elsewhere).

**File:** apps/web/vite.config.ts:14
**Invariant violated / concern:** Defense-in-depth: trusted-origin / CSRF protection on credential endpoints should not be silently disabled.
**Detail:** The dev proxy sets `proxyReq.setHeader('Origin', apiTarget)` for every `/api` request (vite.config.ts:14-18). better-auth validates request Origin against `trustedOrigins` (packages/auth/src/index.ts:202, default `http://localhost:3000`). With the proxy forcing Origin to the API target, the origin check always passes regardless of the true requester origin during dev. This is dev-server-only (the production build is served statically and does not use this proxy), so it is not a production exposure. Worth documenting so it is not mistaken for production behavior.
**Verification:** Confirmed at INFO. apps/web/vite.config.ts:14-17 registers `proxy.on("proxyReq", ...)` forcing the Origin header to `apiTarget` (default `http://localhost:3000`, line 4) on every proxied request. packages/auth/src/index.ts:202 sets `trustedOrigins: readTrustedOrigins(env)`, defaulting (lines 368-369) to `http://localhost:3000` — so the rewritten Origin always matches and the better-auth origin check always passes. apps/web/package.json:8 (`"dev": "vite --host 0.0.0.0"`) plus docs/operations/dev-environment.md:8,12 confirm the LAN-dev convention binds the proxying dev server to all interfaces. **Suggested fix:** Do not run the Vite dev server with `--host` on untrusted networks, or drop the forced Origin rewrite and instead add the dev origin (e.g. `http://<lan-ip>:5173`) to `JARVIS_AUTH_TRUSTED_ORIGINS` so better-auth's check is exercised honestly rather than bypassed.

#### [INFO] Rate-limit key falls back to attacker-influenced first-hop XFF when JARVIS_TRUST_PROXY is enabled (accepted spec tradeoff)
**File:** apps/api/src/server.ts:64
**Invariant violated / concern:** Rate limiting must remain effective against per-request IP rotation.
**Detail:** `keyGenerator` is `request.ip` (server.ts:64). With `JARVIS_TRUST_PROXY` unset (default), `request.ip` is the socket peer and XFF is ignored — the api-rate-limit.test.ts:157 'C1 regression' test confirms XFF spoofing cannot bypass the throttle in this mode. When an operator sets `JARVIS_TRUST_PROXY` (legitimate reverse-proxy deploy), `request.ip` resolves from the XFF chain; a client that can reach the API directly (or inject XFF through a misconfigured proxy) could rotate the leftmost hop to evade the per-IP login limit. The rate-limiting spec (2026-06-09-p1-rate-limiting.md, Decision C) explicitly accepts first-hop-XFF for proxied deployments, so this is a documented, intentional tradeoff and not a regression — recording it only so the trust boundary is explicit.
**Verification:** Confirmed at INFO. apps/api/src/server.ts:64 — `keyGenerator: (request) => request.ip`; server.ts:51 — `trustProxy: !!process.env.JARVIS_TRUST_PROXY`, with comments (lines 49-50/61-63) stating that when set, Fastify resolves `request.ip` from the XFF chain, otherwise XFF is ignored. tests/integration/api-rate-limit.test.ts:157-190 — "XFF-spoof per-request does NOT bypass throttle" injects three POSTs with rotating `x-forwarded-for` from the same `remoteAddress` and asserts the third is 429, confirming spoof-proofing in the default mode. docs/superpowers/specs/2026-06-09-p1-rate-limiting.md:59-64 (Decision C) explicitly accepts first-hop-XFF keying for proxied deployments and documents the `trustProxy` assumption. Rationale: technically accurate that the enabled `!!` boolean form takes the leftmost client-controlled XFF hop (no hop count or trusted-CIDR allowlist), but it is gated behind an explicit operator opt-in for a documented single-instance self-host with the trust boundary spelled out in the approved spec, and the safe default is proven by test — a legitimate trust-boundary observation, not an exploitable-now invariant break. INFO is right.
**Suggested fix:** No change required for the documented single-instance/self-host model. If a multi-hop proxy chain is ever introduced, set Fastify `trustProxy` to the proxy hop count (not boolean `true`) so only the actual edge proxy's appended IP is trusted as the key.

#### [INFO] No global Fastify error handler — unknown errors re-thrown by handleRouteError fall through to Fastify's default 500
**File:** apps/api/src/server.ts:67
**Invariant violated / concern:** Secrets/internal detail must never escape in error responses; an explicit error boundary should normalize all unhandled errors.
**Detail:** `createApiServer` wires routes inside `server.after()` (lines 67-107) but never calls `server.setErrorHandler(...)`. Module route handlers funnel unknown errors through `handleRouteError`, which re-throws errors it does not recognize (confirmed in chat/settings/connectors route modules). A re-thrown unknown error therefore reaches Fastify's built-in default handler. Fastify's default serializer omits stack traces from the JSON body and logs them server-side, so this is not an active leak today — hence INFO, not a finding of disclosure. However, relying on the framework default rather than an explicit boundary is fragile: a future error subclass with an enumerable message/detail (e.g. a driver error echoing a query fragment) could surface in the 500 body. The first auditor noted the rate-limit and proxy posture but did not observe the absence of a central error boundary.
**Verification:** Confirmed at INFO. apps/api/src/server.ts:67-107 — routes registered inside `server.after()` with no `server.setErrorHandler(...)` call; grep for `setErrorHandler` across packages/ and apps/api/ returns zero hits. packages/module-registry/src/index.ts:193-200 — `registerBuiltInApiRoutes` calls `module.registerRoutes?.(...)` with no wrapping try/catch. `handleRouteError` re-throws unrecognized errors at packages/chat/src/routes.ts:287, packages/settings/src/routes.ts:540, packages/connectors/src/routes.ts:418 — each matches an allowlist of known `error.message` strings, maps them to safe 401/403/400 codes, then `throw error` for anything else. Fastify ^5.6.2 (apps/api/package.json:17). Rationale: structurally exactly as described and verified; Fastify 5's default error handler does not serialize `error.message` into the JSON body for 500-class errors (generic "Internal Server Error" body, real error logged server-side), so a bare re-thrown driver/DB error does not leak its message on the response path today. A defense-in-depth/hardening observation about depending on a framework default — fragile if a future error subclass carries an enumerable 4xx `statusCode` whose message echoes a query fragment — not an active invariant break. Confirmed at INFO.
**Suggested fix:** Register an explicit `server.setErrorHandler` that maps validation errors to 400, known domain errors to their status, and everything else to a generic 500 with no message/detail, while logging the full error server-side. This makes the no-secret-in-error-body guarantee a property of the server rather than of each module's `handleRouteError`.

#### [INFO] chat_memory_facts UPDATE policy has USING but no WITH CHECK (downgraded from LOW — implicit Postgres fallback holds)
**File:** packages/memory/sql/0041_memory_facts.sql:37
**Invariant violated / concern:** Private by default — owner-only rows must not be re-owned; FORCE-RLS UPDATE policies on owner-scoped tables conventionally state both USING and WITH CHECK.
**Detail:** The `chat_memory_facts_update` policy (lines 36-38) declares `FOR UPDATE USING (owner_user_id = app.current_actor_user_id())` but omits a WITH CHECK clause. The original finding posited that, lacking WITH CHECK, an UPDATE could reassign `owner_user_id` to another user. The skeptic refuted that exploit mechanism (see Verification) but the consistency gap is real. The app grants UPDATE on this table to both `jarvis_app_runtime` and `jarvis_worker_runtime` (lines 44-45). The INSERT policy (line 34) correctly uses WITH CHECK. The first auditor did not review the memory module's RLS policy completeness.
**Verification:** Downgraded LOW → INFO. packages/memory/sql/0041_memory_facts.sql:36-38 confirms `FOR UPDATE USING (...)` with no WITH CHECK; grants to both runtime roles at lines 44-45; the INSERT policy at line 34 does use WITH CHECK. The single repository write path `ChatMemoryFactsRepository.updateFactImportance` (packages/memory/src/facts-repository.ts:93-103) sets only `importance`/`updated_at` and filters by `id`; its sole caller is packages/chat/src/routes.ts:220 wrapped in `withDataContext`; no other write path exists. Rationale: the claimed exploit is refuted — per documented PostgreSQL behavior, when WITH CHECK is omitted on a FOR UPDATE policy the USING expression is automatically applied as the check against the post-update (new) row, so an attempt to set `owner_user_id` to another user's id is evaluated against `owner_user_id = app.current_actor_user_id()`, fails, and is rejected by RLS; the row cannot be re-owned even via a hypothetical injection foothold. What remains is a stylistic/consistency gap: the codebase overwhelmingly states WITH CHECK explicitly on FOR UPDATE policies (sibling memory tables 0030:56-58, 0032:54-56; chat 0035/0036; shares 0017; ai/tasks/briefings/calendar/email/connectors/structured-state). Worth an explicit WITH CHECK for clarity, but not a LOW-severity security weakness — hence INFO. (Note: packages/chat/sql/0042_chat_memory_settings.sql:25 shares the identical omission and would warrant the same INFO note.)
**Suggested fix:** Add `WITH CHECK (owner_user_id = app.current_actor_user_id())` to the `chat_memory_facts_update` policy in a NEW migration file (never edit the applied 0041), matching the INSERT policy's check, so the policy states its post-update constraint explicitly rather than relying on the implicit Postgres fallback. Apply the same to `chat_memory_settings.sql:25`.

## Refuted (audit trail)

- **OAuth authorize endpoint accepting a Google client secret is not rate-limited (its /complete sibling is)** — packages/connectors/src/routes.ts:60 — Refuted by the skeptic: the raw code facts are accurate (`/authorize` has no `config.rateLimit`; `/complete` sets `max=oauthMax`), but the two stated abuse vectors do not hold. (1) "Unbounded pending-authorization rows" is false — `upsertGooglePending` (repository.ts:162-182) DELETEs all pending rows for the provider then INSERTs exactly one, so there is at most one pending row per user; hammering only overwrites it. (2) "Abuse the upstream Google authorize URL generation" is false — `buildAuthUrl` (oauth.ts:57-73) is pure local URL-string construction with NO network call; only `/complete` makes a network round-trip, which is precisely why the spec throttles it. The rate-limiting spec (2026-06-09-p1-rate-limiting.md) scopes the throttled set extensionally to `/api/auth/*` and `POST /api/connectors/google/complete` (Context 11-17, Goals 22-27, Non-Goal 31, Approach 80-83); `/authorize` was deliberately out of slice scope and is authenticated/per-user, not the pre-auth abuse target. At most a LOW/INFO defense-in-depth observation, not a MED; a future hardening slice could add an authenticated per-user throttle if desired.
