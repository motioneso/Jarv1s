#### [HIGH] User deletion never removes the user's vault files — private notes orphaned on disk after account delete
**File:** scripts/delete-user-data.ts:115
**Invariant violated / concern:** Private by default / complete erasure — a deleted user's private data (vault notes) must not survive deletion; the deletion path must cover encrypted credentials AND vault files.
**Detail:** deleteUserData() only opens a pg.Client and runs `DELETE FROM app.users WHERE id = $1` (line 115), relying entirely on FK ON DELETE CASCADE. That cascade is verified to remove the DB-resident secrets (app.connector_accounts, app.connector_oauth_pending, app.ai_provider_configs all declare `REFERENCES app.users(id) ON DELETE CASCADE` — packages/connectors/sql/0044_google_unified_connection.sql:33, packages/ai/sql/0013_ai_module.sql:40), so encrypted credentials ARE erased. But the script never imports @jarv1s/vault and never touches the filesystem. The user's vault directory `/data/vaults/<userId>/` (created with mode 0o700 in packages/vault/src/vault-context.ts:31-32 and holding the user's ingested private notes) is left fully intact on disk after the account row is gone. There is no whole-vault purge utility anywhere in the tree (only deleteVaultFile for single files, packages/vault/src/vault-ops.ts:28). Result: account deletion is incomplete — all of a user's private note content remains readable on the host, now orphaned (no DB row maps to it), defeating the deletion guarantee.
**Verification:** confirmed (HIGH). scripts/delete-user-data.ts:115 — sole mutation is `DELETE FROM app.users WHERE id = $1::uuid`; imports (lines 1-7) are only node:crypto, node:path, node:url, pg, @jarv1s/db — no @jarv1s/vault, no node:fs. Vault content is per-user keyed by userId (vault-context.ts:31, default base /data/vaults per vault-config.ts:1, mode 0o700) and holds ingested private markdown notes (packages/memory/src/ingest.ts:36 reads via readVaultFile). Only deletion primitive is single-file deleteVaultFile (vault-ops.ts:28-31); grep finds no recursive whole-vault purge. DB-resident secrets DO cascade and tests/integration/release-hardening.test.ts:102-103 assert userAConnectorRows=0 / userAAiProviderRows=0 post-delete, but the deletion test (85-109) asserts only DB row counts — no filesystem assertion. Rationale: real, concrete violation of a deletion invariant with a clear path; HIGH not CRIT because it is an operator-run maintenance script (not a live runtime exploit) leaking plaintext notes on the local host rather than the AES-encrypted secrets (which ARE erased).
**Suggested fix:** After the DB COMMIT succeeds (so a failed DB delete never deletes files first), recursively remove the user's vault directory via a vault-owned helper, e.g. `await rm(join(getVaultBaseDir(), userId), { recursive: true, force: true })` guarded behind the same --execute/--confirm-user-id check. Order DB-delete-then-vault-purge so a partial failure leaves files (recoverable) rather than orphaning DB rows; log the purge in the admin_audit_events metadata. Also add a corresponding vault count to the dry-run report so operators see file-cleanup scope.

#### [MED] Secret-encryption keyring silently falls back to a hardcoded dev default unless NODE_ENV is exactly "production"
**File:** packages/db/src/keyring.ts:29
**Invariant violated / concern:** Connector/AI secrets are AES-256-GCM encrypted at rest with an operator-supplied key — real credentials must never be encrypted under a publicly-known constant.
**Detail:** resolveKeyring() only throws `${keyEnvVar} is required in production` when `env.NODE_ENV === 'production'` (line 29). For any other value (unset, 'development', 'staging', 'prod', a typo, or a container that doesn't set NODE_ENV), a missing JARVIS_AI_SECRET_KEY / JARVIS_CONNECTOR_SECRET_KEY silently resolves to the hardcoded devDefault strings `jarv1s-development-ai-secret` / `jarv1s-development-connector-secret` (passed at packages/ai/src/crypto.ts:88 and packages/connectors/src/crypto.ts:90), SHA-256'd into the AES key (keyring.ts:34). Anyone with the public source can derive that key and decrypt every connector token / API credential. The guard is correct in spirit but hinges on a single env value that is easy to omit in a real deployment, so at-rest encryption can degrade to obfuscation without any startup error.
**Verification:** confirmed (MED). keyring.ts:27-34 gates the throw solely on the literal `env.NODE_ENV === "production"`; otherwise `rawCurrentSecret = currentSecret ?? devDefault` is SHA-256'd into the AES-256-GCM key. Hardcoded public devDefaults are passed in at ai/crypto.ts:88 and connectors/crypto.ts:90, used by AiSecretCipher.encryptJson / ConnectorSecretCipher.encryptJson. Real runtime instantiation uses live process.env by default (ai/routes.ts:99, connectors/routes.ts:51); repo-wide grep found no centralized env validation — this single NODE_ENV check is the only guard, and the same fail-open pattern is mirrored for BETTER_AUTH_SECRET (auth/index.ts:358-365). Rationale: fail-open misconfiguration / missing-by-default hardening gap — the invariant degrades only under operator misconfiguration (not on the canonical NODE_ENV=production path), which is a defense-in-depth gap, matching MED.
**Suggested fix:** Fail closed: require the real key whenever NODE_ENV !== 'development' (allowlist the safe case rather than denylisting only 'production'), or require an explicit JARVIS_ALLOW_DEV_SECRET_KEY=1 opt-in before ever using devDefault. At minimum emit a loud startup warning when the dev default is selected so operators notice.

#### [MED] Encryption key derived by raw SHA-256 of passphrase — no salt, no KDF (offline-bruteforce exposure for connector/AI secrets at rest)
**File:** packages/db/src/keyring.ts:34
**Invariant violated / concern:** Secrets never escape; connector/AI secrets are AES-256-GCM encrypted at rest (the at-rest protection must be robust, not just present).
**Detail:** resolveKeyring derives the AES-256-GCM key as createHash("sha256").update(rawCurrentSecret).digest() (line 34, and again for retired keys at line 48). There is no salt and no password-based KDF (no PBKDF2/scrypt/argon2). The encryption key is therefore a deterministic, unstretched function of the JARVIS_*_SECRET_KEY env value. Two consequences: (a) if the configured secret is a human-chosen passphrase rather than 32 bytes of CSPRNG output, an attacker who exfiltrates the encrypted_secret/encrypted_credential ciphertext (jsonb columns are AES-256-GCM with the auth tag and IV stored alongside) can mount an offline dictionary/brute-force attack at SHA-256 speed with zero work-factor; (b) identical passphrases across deployments yield identical keys (no per-deployment salt). This is distinct from the dev-default fallback finding (which only triggers when the env var is unset and NODE_ENV!=='production'): the KDF weakness applies even when a production operator correctly sets JARVIS_AI_SECRET_KEY / JARVIS_CONNECTOR_SECRET_KEY. The ciphers themselves (packages/ai/src/crypto.ts, packages/connectors/src/crypto.ts) are sound AES-256-GCM; the weakness is solely in key derivation.
**Verification:** confirmed (MED). keyring.ts:34 derives the AES key as a single unsalted SHA-256 of the env secret (same at keyring.ts:48 for retired keys); grep for pbkdf2|scrypt|argon|hkdf across packages/ returned no hits. The ciphers are sound AES-256-GCM with random 12-byte IV + stored auth tag (ai/crypto.ts:18-31, connectors/crypto.ts:18-31), so the weakness is isolated to derivation. The derivation runs even with the env var correctly set (`currentSecret ?? devDefault`), distinct from the dev-default path; no per-deployment salt means identical passphrases yield identical keys. Rationale: MED not higher because exploitability is doubly conditional — it needs a low-entropy human passphrase (the rotation runbook docs/operations/secret-key-rotation.md:48 already instructs `openssl rand -base64 32` = 256 bits, making SHA-256 an adequate identity transform) AND prior ciphertext exfiltration (a DB compromise). A robustness / defense-in-depth gap, not an invariant broken on a runtime path; the mitigating guidance is buried in a rotation runbook with no high-entropy warning at the config site.
**Suggested fix:** Either (a) require the env var to be 32 bytes of base64/hex CSPRNG material and use it directly (validate length, reject short/low-entropy values at boot), or (b) run the passphrase through a salted KDF (scrypt/argon2id) with a per-deployment salt persisted alongside the keyring config. Document that the secret must be high-entropy. Add a boot-time assertion that rejects obviously weak keys in production.

#### [LOW] Google chat adapter places the per-user API key in the request URL query string (?key=<secret>)
**File:** packages/ai/src/adapters/http-api.ts:100
**Invariant violated / concern:** Secrets never reach logs (Connector/AI credentials, auth tokens, password hashes, and session tokens never reach frontend responses, logs, pg-boss job payloads, user exports, or AI prompts).
**Detail:** In buildRequest, the 'google' branch constructs url = `https://generativelanguage.googleapis.com/v1beta/models/${encodeURIComponent(modelId)}:generateContent?key=${this.apiKey}` (line 100), then fetches it (line 46). The anthropic and openai-compatible branches correctly carry the key in headers (x-api-key line 74, authorization Bearer line 90), and generateChat is explicitly careful to keep the key out of thrown errors (lines 53-54 throw only `HTTP ${status}`). But a secret embedded in a URL is fundamentally more leak-prone than one in a header: full request URLs routinely appear in reverse-proxy/access logs, undici/Node error and debug output, and any future request-logging middleware — none of which the header-based providers expose. This contradicts the spirit of the at-no-point-in-logs invariant that the surrounding code visibly tries to honor. The in-URL key placement is a latent code defect that becomes a live leak the moment the adapter is wired. Google's API supports an `x-goog-api-key` header that would avoid the URL.
**Verification:** downgraded to LOW (originally MED). http-api.ts:99-100 confirms the google branch embeds `?key=${this.apiKey}`, while anthropic (line 74) and openai-compatible (line 90) use headers; error path is scrubbed (lines 53-54 throw only `HTTP ${status}`, with a "Never include the API key in error messages" comment). No production caller — only the re-export in chat-adapter.ts:1-3 (labeled dead-code substrate for a future tie-in) and tests/unit/ai-http-api.test.ts (which at :142-146 deliberately asserts `key=sk-test-google` IS in the URL, confirming intentional design). Rationale for downgrade: the asymmetry and the header alternative are real, but the leak is unexploitable today on every axis (zero production callers, no URL-emitting logging in generateChat, scrubbed error path) and only materializes if (a) the adapter is wired into a worker AND (b) a URL-logging layer is later added — a latent hardening item on dead code, fitting LOW better than MED.
**Suggested fix:** Send the key via the `x-goog-api-key` request header (supported by the Generative Language API) instead of the `?key=` query parameter, matching the header-based handling already used for anthropic/openai. This keeps the secret out of any URL that could be logged by proxies or Node's HTTP stack.

#### [LOW] Google token-endpoint error body is echoed to the API client and logs via an unclassified 500
**File:** packages/connectors/src/oauth.ts:110
**Invariant violated / concern:** Secrets never escape to responses/logs; upstream provider errors should be sanitized before surfacing.
**Detail:** postToken() throws `new Error(\`Google token endpoint returned ${response.status}: ${detail}\`)` where detail = response.text() of Google's OAuth token endpoint (oauth.ts:110-111). This is a plain Error, not GoogleConnectError, so connectors handleRouteError (packages/connectors/src/routes.ts:392-418) matches none of its branches and re-throws at line 418 — Fastify returns a 500 carrying the message, and the default logger (server.ts:48, logger:true) logs it. Google's documented token-error bodies (`invalid_grant`/`invalid_client` with error_description) do NOT echo back the submitted client_secret, so this is not a confirmed secret leak; but it forwards a verbose, attacker-influenceable upstream body to the client and into logs, and conflates an upstream 4xx with a server 500.
**Verification:** confirmed (LOW). oauth.ts:109-111 throws a plain Error carrying Google's raw response body; google-connection.ts:81 (exchangeCode) and :120 (refreshAccessToken) call postToken with no surrounding try/catch, so it propagates unwrapped through routes.ts:93-97 to handleRouteError (routes.ts:392-418), which only special-cases GoogleConnectError/HttpError/four exact strings and re-throws unrecognized Errors at line 418. server.ts:48 sets `logger: true` and grep found no setErrorHandler, so Fastify's default handler returns a 500 with the message and logs the throw. The posted request body DOES include client_secret (oauth.ts:84,97-98), but `detail` is Google's *response* text, which per documented error shapes does not reflect the submitted secret. Rationale: genuine hygiene / defense-in-depth gap (unsanitized upstream body forwarded to client + logs, 4xx-as-500) but NOT a confirmed secret leak, and the exposure is bounded (attacker only influences the pasted code; refresh runs server-side) — correctly LOW.
**Suggested fix:** Wrap the failure in GoogleConnectError with a generic operator-facing message (e.g. "Google rejected the authorization — re-check the client credentials") and log the upstream detail at debug level only, never in the HTTP response. This also gives the caller a 400 instead of a 500.

Rationale: NOT a security/data-integrity defect — every omitted table cascades correctly (no orphans, no RLS bypass, no secret leak); impact is confined to the accuracy of the operator audit trail (a data-subject-deletion compliance record), a hygiene/observability gap with no exploitable path — correctly LOW.
**Suggested fix:** Add app.memory_chunks, app.memory_links, app.chat_memory_facts, app.commitments, app.entities, app.preferences (and app.connector_oauth_pending) to userScopedCountQueries with their owner_user_id predicates, so the dry-run and the persisted audit event reflect the true deletion blast radius. Consider generating the list from a single canonical registry to prevent future drift as modules add tables.
