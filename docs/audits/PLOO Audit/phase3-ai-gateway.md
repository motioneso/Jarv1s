#### [MED] Gateway input validator silently ignores integer/enum/min-max/additionalProperties constraints declared in tool inputSchema
**File:** packages/ai/src/gateway/input-validation.ts:39
**Invariant violated / concern:** All tool inputs pass validation before dispatch (spec: "input schema-validated by the Gateway").
**Detail:** validateToolInput only checks required-key presence and the five coarse JSON types in JSON_TYPE_OF (string/number/boolean/object/array). It has no handling for: the JSON Schema "integer" type (tasks.list declares priority as {type:"integer",minimum:1,maximum:5}; "integer" is not a key in JSON_TYPE_OF so the check is undefined and the field is accepted unvalidated — any number, including 9.9 or 1e9, passes); "enum" (status/quadrant enums are not enforced); minimum/maximum bounds; and additionalProperties:false (connectors.startGoogleGuidance declares it but extra keys are silently accepted). The validator is the single declared input gate before module dispatch (gateway.ts:61). Today every executable tool handler re-narrows its own inputs and all DB access is parameterized Kysely under a DataContextDb (e.g. packages/tasks/src/tools.ts:24-61), so this is not currently exploitable into SQL/shell/fs — hence MED, not HIGH. But it means the gateway provides far weaker input enforcement than the manifests advertise, so a future write/destructive tool author who trusts "input is already validated against inputSchema" (the contract stated in module-sdk ToolExecute doc, packages/module-sdk/src/index.ts:29-33) could receive out-of-bounds or wrong-typed values.
**Verification:** Confirmed (MED). JSON_TYPE_OF defines only string/number/boolean/object/array (input-validation.ts:10-16); the property loop reads only `declared.type` and at line 45 does `if (check && !check(...))` — when declared.type is "integer", the lookup is undefined, `check` is falsy, and the field passes unvalidated. enum/minimum/maximum/additionalProperties are never referenced. tasks.list (manifest.ts:240-251) declares priority integer 1–5 plus status/quadrant enums; connectors.startGoogleGuidance declares additionalProperties:false (manifest.ts:135) — none enforced. validateToolInput is the sole input gate before dispatch (gateway.ts:60-61); mcp-transport.ts:85 passes params.arguments straight through with no upstream validator (no AJV in path). Not exploitable today: handlers re-narrow over parameterized Kysely under DataContextDb, and the lone write tool (tasks.updateStatus) is declaration-only with no execute handler. Rationale: real defense-in-depth gap and a genuine mismatch with the advertised ToolExecute contract, creating forward risk — correctly MED.
**Suggested fix:** Either replace validateToolInput with a real JSON Schema validator (ajv) at the gateway boundary, or extend it to handle integer (typeof number && Number.isInteger), enum membership, minimum/maximum, and additionalProperties:false. At minimum, document loudly in module-sdk that handlers must not assume full schema validation.

#### [MED] REST tool-invoke route bypasses validateToolInput, violating the SDK "input is already validated" contract that the gateway upholds
**File:** packages/ai/src/routes.ts:418
**Invariant violated / concern:** All tool inputs pass validation before dispatch (KEY QUESTION 3); ToolExecute handlers may assume input is validated against inputSchema (module-sdk/src/index.ts:27).
**Detail:** There are two execution surfaces for assistant tools. The MCP gateway (packages/ai/src/gateway/gateway.ts:61) calls validateToolInput(found.tool.inputSchema, rawInput) before dispatch. The sibling REST route POST /api/ai/assistant-tools/:name/invoke does NOT: at packages/ai/src/routes.ts:418 it passes `body.input ?? {}` straight into manifestTool.execute(...) with no schema check at all. parseInvokeAssistantToolBody (routes.ts:504) only asserts the wrapper is a JSON object — it never validates the inner `input` against the tool's declared inputSchema (required keys, declared types). The module-sdk explicitly documents ToolExecute as receiving input "already validated against inputSchema" (packages/module-sdk/src/index.ts:27), so this route breaks the contract every executable read tool is written against. It is not exploitable as injection today only because the currently-registered read executes (packages/tasks/src/tools.ts) route everything through Kysely-parameterized queries / in-memory .filter() under an RLS-scoped scopedDb; the moment a read tool trusts input shape (e.g. assumes a required field is present, or a numeric field is a number) it inherits unvalidated, attacker-shaped input on this path. The first auditor reviewed only the gateway path; this second egress was not covered.
**Verification:** Confirmed (MED). Gateway path validates (gateway.ts:61, used by runHandler:96 and confirmAndRun:150). REST path does NOT (routes.ts:417-423 invokes `manifestTool.execute!(scopedDb, body.input ?? {}, …)` with no validateToolInput call; validateToolInput is not even imported into routes.ts). parseInvokeAssistantToolBody only does requiredJsonObject, which checks only non-null/non-array object (routes.ts:504-514, 626-632); the Fastify body schema types `input` as `{type:"object", additionalProperties:true}` (shared/src/ai-api.ts:504-510, 189-192), so the declared inputSchema is never enforced at the framework layer either. Not exploitable today: current read executes (tasks/src/tools.ts:24-32,57,69,147) cast input with TS `as` (runtime-erased) but route through parameterized Kysely `.where(...)` / in-memory `.filter()` under RLS-scoped scopedDb — bad input yields empty/NaN results, not injection or cross-user access. Rationale: genuine contract/defense-in-depth gap that becomes a latent bug the moment a future read tool trusts input shape; also note the same surface skips validateToolInput for write/destructive tools after confirmation (routes.ts:417 reached for all risks), so slightly broader than read-only — still MED.
**Suggested fix:** Call validateToolInput(manifestTool.inputSchema, body.input ?? {}) in the REST invoke route before manifestTool.execute, mirroring gateway.callTool, and return 400 on ToolInputValidationError. Better: route both surfaces through a single shared execute-with-validation helper so the two paths cannot drift.

#### [LOW] tools/list is not actor-scoped (passes empty actorUserId) — latent over-exposure once per-user module enablement lands
**File:** packages/ai/src/gateway/gateway.ts:47
**Invariant violated / concern:** A user should only see/reach tools for modules enabled for them (the ActiveModulesResolver enablement seam, types.ts:3-11).
**Detail:** listTools() calls this.executableTools("") with a hardcoded empty actorUserId, while callTool() correctly passes the verified actorUserId to executableTools(actorUserId) (gateway.ts:54). The MCP transport tools/list handler invokes gateway.listTools() with no user context (packages/chat/src/mcp-transport.ts:74). Today this is harmless because the wired resolver (resolveActiveModules: deps.listModuleManifests, module-registry/src/index.ts:148) ignores actorUserId and returns all built-in modules. But the documented purpose of the resolver is to become per-user ("when per-user module enable/disable ships (#30), this resolver reads it", types.ts:6-9). When that lands, tools/list would advertise the full tool surface to every user while callTool would correctly filter — a list/enforce mismatch and an information-disclosure of disabled tool names.
**Verification:** Confirmed (LOW). listTools() hardcodes `executableTools("")` (gateway.ts:46-48) while callTool derives actorUserId from the verified token and calls executableTools(actorUserId) (gateway.ts:50-54); executableTools forwards the id to resolveActiveModules (gateway.ts:172-173). The MCP transport invokes the unscoped path (mcp-transport.ts:70-75). The resolver seam is documented to become per-user (types.ts:3-11). Today the wired resolver is actor-agnostic: getBuiltInModuleManifests returns BUILT_IN_MODULES.map((m) => m.manifest) ignoring its argument (module-registry/src/index.ts:181-183; wired at server.ts:102, :148), so listTools("") and listTools(actorUserId) are identical now. Rationale: inert today, becomes a list/enforce divergence only after the #30 enablement seam lands, and even then the ceiling is disclosure of disabled tool names/descriptions — callTool re-resolves with the real actorUserId and rejects tools not in the active set, so no unauthorized execution. Correctly LOW.
**Suggested fix:** Thread the verified token's actorUserId into the tools/list path: have the MCP transport pass the token to a gateway.listToolsForToken(token) that verifies and calls executableTools(actorUserId), so list and call resolve the same per-user set.

#### [LOW] REST tool-invoke route lacks the gateway's handler-error scrubbing, so a tool execute() throw leaks its message verbatim in the 500 response
**File:** packages/ai/src/routes.ts:434
**Invariant violated / concern:** Secrets / internals never escape via responses; handler throws must not leak internal detail (the gateway treats this as an invariant).
**Detail:** The gateway deliberately scrubs handler throws: AssistantToolGateway.runHandler (packages/ai/src/gateway/gateway.ts:99-102) catches any error from found.execute and returns a generic `Tool <name> failed`, with the comment "never leak internals/secrets from a handler throw". The REST invoke route has no equivalent: when manifestTool.execute (routes.ts:418) throws, the catch at routes.ts:434 calls handleRouteError, which only special-cases HttpError / known auth strings and otherwise re-throws (routes.ts:789). No custom Fastify error handler is registered anywhere (no setErrorHandler in apps/api/src or packages), so Fastify's default handler returns a 500 whose body includes the thrown Error.message. A module read-tool whose error message embeds internal context would surface it to the caller on this path, whereas the same throw is scrubbed on the gateway path. Asymmetry between the two egress surfaces; current read tools don't embed secrets in messages, hence LOW.
**Verification:** Confirmed (LOW). runHandler's catch returns a generic `Tool ${name} failed` with the "never leak internals/secrets" comment (gateway.ts:99-102). The REST read-risk path invokes execute inside a try (routes.ts:417-435) and routes throws to handleRouteError, which special-cases only HttpError, three exact auth strings, and FK/RLS/duplicate-key substrings, then `throw error` at routes.ts:789. No setErrorHandler is registered (server.ts:47-52, grep across apps/ and packages/ returns zero hits), so Fastify's default handler serializes a 500 body including Error.message. Asymmetry confirmed. Severity: current read executes (tasks/src/tools.ts:19-88, calendar/email list-visible) throw only DB/framework/repository errors, not custom secret-bearing messages, and RLS errors are caught and genericized to a 400 — so what leaks today is internal/driver text, not a proven secret. Correctly LOW. (Note: the same default-handler leak applies to any unhandled throw on this route, e.g. body-parse, broadening the surface but reinforcing the finding.)
**Suggested fix:** Wrap manifestTool.execute in the REST route in a try/catch that returns a generic blocked/failed invocation (reusing serializeAssistantToolInvocation with a tool_failed reason) exactly as runHandler does, or register a global Fastify error handler that strips Error.message from 5xx responses.

#### [LOW] Resolving a pending action via the AI module's REST resolve route does not unblock the gateway's in-memory confirmation waiter
**File:** packages/ai/src/routes.ts:321
**Invariant violated / concern:** The confirm-and-run bridge: a confirmed action should unblock the waiting tool call (correctness of the human-in-the-loop confirmation gate).
**Detail:** Two endpoints can resolve a pending ai_assistant_action_requests row. POST /api/chat/action-requests/:id/resolve (packages/chat/src/routes.ts:126) calls gateway.resolveActionRequest, which both persists the resolution AND calls confirmations.resolve(actionRequestId, status) (packages/ai/src/gateway/gateway.ts:85) to release the blocked callTool. The AI module's own POST /api/ai/assistant-actions/:id/resolve (packages/ai/src/routes.ts:321) only performs the DB update via repository.resolveAssistantAction — it has no reference to the ConfirmationRegistry (that route's deps never construct one). So a client that confirms a gateway-blocked tool call through the ai route flips the DB row to confirmed but leaves the gateway's awaitResolution waiter hanging until confirmTimeoutMs (150s), after which the call is reported denied/timed-out despite a recorded confirmation. No security breach (both routes are RLS-scoped owner-only via the policies in packages/ai/sql/0016_ai_assistant_actions.sql), but the two resolve surfaces diverge in behavior and the DB/runtime state can disagree.
**Verification:** Confirmed (LOW). Two resolve surfaces on the same table diverge in side-effects: the chat route calls gateway.resolveActionRequest, which persists via repository.resolveAssistantAction AND calls confirmations.resolve(actionRequestId, status) (gateway.ts:79-85); the AI route calls only repository.resolveAssistantAction (routes.ts:321-323) — a bare DB UPDATE — and its dependency interface AiRoutesDependencies (routes.ts:51-57) carries no ConfirmationRegistry/gateway. The waiter is the only release mechanism (confirmation-registry.ts:15-34; otherwise "timeout"), the registry is owned solely by the chat gateway with the 150s cap (chat/src/routes.ts:70, :73-81; blocked call awaits at gateway.ts:131-148). No security breach: RLS owner-only SELECT/INSERT/UPDATE via `owner_user_id = app.current_actor_user_id()` (0016_ai_assistant_actions.sql:113-139) and the UPDATE is guarded `.where("status","=","pending")` (repository.ts:365). Rationale: real correctness/divergence bug — DB shows confirmed while runtime times out — but no exploit and on a secondary surface; worst case is a delayed/incorrect "timed out / denied" to the model for an approved call. Correctly LOW.
**Suggested fix:** Either route the gateway-confirmation flow exclusively through the chat resolve endpoint (and document the ai route as config-only / non-gateway), or give the ai resolve route access to the same ConfirmationRegistry so it also calls confirmations.resolve after a successful DB update.

#### [INFO] Per-session MCP tokens have no independent time-to-live; only bounded by engine idle-reap
**File:** packages/ai/src/gateway/session-tokens.ts:19
**Invariant violated / concern:** Identity comes from a short-lived per-session token (spec 4.2: "short-lived per-session token").
**Detail:** SessionTokenRegistry stores {actorUserId, chatSessionId} in an in-memory Map with no expiry timestamp. verify() succeeds for any token still in the map; tokens are only removed by revoke()/revokeBySessionId(), which the chat-session-manager calls on /clear, provider-switch, and idle reap (packages/chat/src/live/chat-session-manager.ts:275,290,332). There is no wall-clock TTL on the token itself. As long as the live engine session keeps getting activity (lastActivity refreshed each turn, manager.ts:256) it is never idle-reaped, so the minted token can remain valid indefinitely. The token is injected into CLI launch commands / config (cli-chat-engine.ts:214-216,246-247) and, for Codex, appears in process args (acknowledged tradeoff comment at cli-chat-engine.ts:238-240). A long-lived process-visible bearer token is a wider exposure window than "short-lived" implies.
**Verification:** Downgraded LOW → INFO. The technical mechanics are all real: SessionIdentity (session-tokens.ts:3-6) is {actorUserId, chatSessionId} with no expiry; verify() (28-34) returns identity for any token still in the Map; removal only via revoke()/revokeBySessionId(), driven by lifecycle (clear/switchProvider/reapIdle at chat-session-manager.ts:275,290,332), and runTurn refreshes lastActivity each turn so an active engine is never reaped. Injection confirmed (Claude --mcp-config Bearer at cli-chat-engine.ts:216; Codex env prefix at :238-241 with explicit accepted-tradeoff comment). HOWEVER the cited concern misreads the spec: spec 4.2 reads "mints a short-lived per-session token … and revokes it when the engine is reaped (lifecycle-scoped, not a fixed TTL that could expire mid-conversation)" — the design deliberately DEFINES "short-lived" as lifecycle-scoped and explicitly REJECTS a wall-clock TTL. Residual risk is minimal: the token is an in-memory (restart-volatile) handle mapping only to its own owner's {actorUserId, chatSessionId}, and the gateway runs every tool under withDataContext (gateway.ts:51-52), so RLS bounds it to that user's own data even if leaked — not a cross-user credential. Rationale: behavior is intended, documented, and RLS-bounded — a benign observation, not a defect; the only genuinely improvable item is the doc-clarity nit that "short-lived" is shorthand for "lifecycle-scoped." Downgraded to INFO.
**Suggested fix:** Add a mint timestamp + maxAgeMs to SessionIdentity and reject in verify() once exceeded (forcing a re-mint on the next launch). Even a generous cap (e.g. matching idleMs or a few hours) bounds the exposure window for the process-visible Codex token.

#### [INFO] Gateway returns module handler output verbatim — no secret-scrubbing layer; correctness depends on each module's serializer
**File:** packages/ai/src/gateway/gateway.ts:98
**Invariant violated / concern:** Secrets never escape: credentials/tokens never reach AI prompts/responses.
**Detail:** runHandler returns { ok: true, data: result.data } passing the module handler's data object straight through to the MCP response and thus into the agent's context (mcp-transport.ts:111-115 JSON.stringifies res.data). The gateway has no output-schema enforcement or secret-redaction step; it trusts each module to return only safe fields. Reviewed all currently-exposed read tools and none surface secrets: email/calendar/notifications/tasks return owner-scoped serialized rows via *.listVisible under RLS (e.g. packages/tasks/src/tools.ts), and connectors.startGoogleGuidance returns static guidance text only (packages/connectors/src/manifest.ts:143-154). The connectors access_token path (packages/connectors/src/google-connection.ts:134) is NOT wired as an assistant tool. The AiRepository keeps encrypted_credential out of its safe queries (repository.ts:400-418) and the credential-bearing selectProviderWithCredential is worker-only, not a tool. So no leak exists today — this is a defense-in-depth observation: there is no gateway-level guard if a future tool's serializer regresses.
**Verification:** Confirmed (INFO). runHandler returns the handler's data object verbatim (gateway.ts:98 `return { ok: true, data: result.data };`); the MCP transport JSON.stringifies it into agent context at packages/chat/src/mcp-transport.ts:110-127, line 113 (finding mis-cited the path as packages/ai/src/gateway/mcp-transport.ts and the line as 111-115 — off-by-one/wrong-package, substance unchanged). No output-schema enforcement or redaction anywhere: outputSchema is only copied into a DTO (assistant-tools.ts:16, gateway.ts:190-191), never validated; the only validator is validateToolInput (input only). No leak today: connectors.startGoogleGuidance returns static text; email/calendar/notifications/tasks read tools call repository.listVisible under RLS through module serializers; AiRepository.safeProviderQuery selects only safe columns plus `encrypted_credential IS NOT NULL AS has_credential` (repository.ts:400-418); credential-bearing selectProviderWithCredential (repository.ts:283-311) has zero callers and is not an assistantTool. Rationale: genuine defense-in-depth observation, not an exploitable break of "secrets never escape" today. Correctly INFO.
**Suggested fix:** Optionally enforce declared outputSchema at the gateway (strip undeclared fields) so a single chokepoint prevents accidental secret/field leakage regardless of module serializer correctness.

#### [INFO] validateToolInput silently ignores the "integer" JSON type, and a live registered tool (tasks.list) relies on it
**File:** packages/tasks/src/manifest.ts:246
**Invariant violated / concern:** Declared input constraints are enforced before dispatch.
**Detail:** Corroborates (does not duplicate) the first auditor's MED about input-validation.ts:39 ignoring enum/min-max/additionalProperties. The concrete live instance: tasks.list declares priority as { type: "integer", minimum: 1, maximum: 5 } (packages/tasks/src/manifest.ts:246). The validator's JSON_TYPE_OF map (packages/ai/src/gateway/input-validation.ts:10-16) has keys for string/number/boolean/object/array but NOT "integer", so the lookup at input-validation.ts:44 yields undefined and the priority field is passed through completely unchecked (any type/value). taskListExecute then does an in-memory equality .filter (packages/tasks/src/tools.ts:36), so a bad value simply matches nothing — harmless today, but it demonstrates the type-coverage gap is reachable from a shipped tool, not hypothetical.
**Verification:** Confirmed (INFO). JSON_TYPE_OF has only string/number/boolean/object/array — no "integer" (input-validation.ts:10-16); the guard `const check = JSON_TYPE_OF[declared.type]; if (check && !check(...)) throw …` passes an unknown type through unchecked, and minimum/maximum/enum/additionalProperties are never inspected (only `required` at 32-37 and scalar `type` at 40-48). Live tool tasks.list declares priority integer 1–5 with execute: taskListExecute (manifest.ts:246, :253), and reaches the validator on the live gateway.callTool path (gateway.ts:50-64). Harmless today: taskListExecute applies a pure in-memory equality filter over an RLS-scoped listVisible result (tools.ts:36, scopedDb via assertDataContextDb at tools.ts:20), so a wrong-typed/out-of-range priority matches zero rows — no exposure, no injection (tagId uses parameterized Kysely .where). Rationale: genuine type-coverage gap reachable from production code, corroborating the input-validation MED; not exploitable, no invariant broken — correctly INFO.
**Suggested fix:** Add an "integer" entry to JSON_TYPE_OF (Number.isInteger) and enforce enum/minimum/maximum, or replace the hand-rolled validator with a real JSON-schema validator as the comment at input-validation.ts:18-22 anticipates.
