#### [MED] SettingsRepository violates the DataContextDb-only invariant — admin/settings DB access runs on a raw root Kysely with no withDataContext / actor GUC
**File:** packages/settings/src/repository.ts:64
**Invariant violated / concern:** DataContextDb only — every repository accepts only a branded DataContextDb handle, never a raw Kysely instance; all RLS-scoped access flows through DataContextRunner.withDataContext so the transaction-scoped actor GUC (app.actor_user_id) is set.
**Detail:** SettingsRepository's constructor takes `private readonly db: Kysely<JarvisDatabase>` (repository.ts:64) and every method (getUserById, listUsers, upsertWorkspaceMembership, upsertResourceGrant, deleteResourceGrant, upsertInstanceSetting, insertAuditEvent, countUsers) runs directly on that root pool. It never calls assertDataContextDb and is never wrapped by DataContextRunner.withDataContext. The route layer constructs it with the raw pool: `new SettingsRepository(dependencies.appDb)` (packages/settings/src/routes.ts:75), and module-registry passes the raw appDb while every OTHER module uses deps.dataContext (packages/module-registry/src/index.ts:97-100 vs the chat/tasks/briefings entries at 113/152/160). Confirmed-by-contrast: tasks/memory/chat/ai/structured-state/connectors/email/notifications/shares repositories all take a per-method DataContextDb and call assertDataContextDb (e.g. packages/tasks/src/breakdown.ts:27-28, packages/memory/src/repository.ts:25). Consequence: every settings/admin query executes as jarvis_app_runtime with app.actor_user_id UNSET, so app.current_actor_user_id() returns NULL (infra/postgres/migrations/0002_app_rls.sql:9-13). The app.users SELECT happens to work because users_app_runtime_select is USING(true) (migration 0045 line 83-87), masking the missing GUC; but the settings surface as a whole sits entirely OUTSIDE the RLS-enforcing wrapper that the rest of the codebase relies on as its defense-in-depth backstop. The only thing standing between a caller and cross-user admin-table writes is the application-layer requireAdmin check (packages/settings/src/routes.ts:331-344) — exactly the 'conventions, not DB-level defense' posture the project's hard invariants forbid.
**Verification:** Downgraded HIGH→MED. Structural facts CONFIRMED: repository.ts:64 takes a raw root pool; every method runs on `this.db`, never asserts the brand, never goes through withDataContext; routes.ts:75 is the only production construction and passes raw appDb; module-registry wires settings via deps.appDb while tasks/chat/briefings use deps.dataContext; data-context.ts:22-39/52-60 confirm the GUC-set + brand-assert mechanism, and 0002_app_rls.sql:9-13 confirms current_actor_user_id() is NULL when the GUC is unset — so the "DataContextDb only / all RLS-scoped access flows through withDataContext" invariant is genuinely violated for this surface. But the HIGH "very likely exploitable" rating is overstated: a grep over infra/postgres/migrations found NO ENABLE/FORCE RLS or CREATE POLICY on workspaces / workspace_memberships / resource_grants / instance_settings / admin_audit_events, so the unset actor GUC removes no row-level protection that would otherwise apply; the one RLS-governed table (app.users) is only ever SELECTed under a USING(true) policy; and all 12 admin handlers are gated by requireAdmin(is_instance_admin). No concrete exploit path — defense-in-depth / invariant-conformance gap, MED not HIGH.
**Suggested fix:** Refactor SettingsRepository to take a DataContextDb per method (or have registerSettingsRoutes resolve AccessContext and run each handler inside dependencies.dataContext.withDataContext, passing the branded handle). At minimum, route the admin writes through withDataContext so the actor GUC is set and the access is consistent with the assertDataContextDb invariant the rest of the modules enforce.

#### [LOW] app.resource_grants (the cross-user sharing-control table) has no row-level security; jarvis_app_runtime holds full DML with only app-layer requireAdmin protecting it
**File:** infra/postgres/migrations/0004_auth_workspaces_settings.sql:86
**Invariant violated / concern:** DB-level defense-in-depth: sharing/authorization-control state must not rely solely on application checks. Private-by-default access decisions are made via app.has_resource_grant() reading resource_grants, so the integrity of resource_grants is security-critical.
**Detail:** 0004 grants SELECT/INSERT/UPDATE on app.workspaces, app.workspace_memberships, app.resource_grants, app.instance_settings to jarvis_app_runtime (line 86-88) and 0005 adds DELETE on workspace_memberships + resource_grants (line 20), but NO migration ever runs ENABLE/FORCE ROW LEVEL SECURITY on these tables — a full-repo grep for `ENABLE ROW LEVEL SECURITY` over infra/postgres/migrations matching these names returns only app.users (0045:51). The release-hardening audit's protectedTables list (scripts/audit-release-hardening.ts:30-45) deliberately omits all four, so this is a known classification for the pure-config tables (workspaces/instance_settings) — defensible since they are admin-gated. The concern is resource_grants specifically: it is the sharing-control table consulted by the SECURITY DEFINER app.has_resource_grant() for every cross-user access decision (infra/postgres/migrations/0002_app_rls.sql:43-63), yet any code path executing as jarvis_app_runtime can INSERT/UPDATE/DELETE grant rows with zero DB backstop — the sole guard is the requireAdmin application check in the settings routes. A future non-admin-gated route, a bug in requireAdmin, or any other app_runtime query touching resource_grants would be able to forge or delete grants and silently widen another user's data exposure.
**Verification:** Downgraded MED→LOW. The physical observation is true (resource_grants has no ENABLE/FORCE RLS; jarvis_app_runtime holds full DML, guarded only by requireAdmin). But the MED rating rests on the claim that resource_grants is the table the access engine reads for cross-user decisions, and that is FALSE in the current tree: the only live app.has_resource_grant() call sites (0002_app_rls.sql:108 rls_probe_items_select; packages/tasks/sql/0003_tasks_module.sql:116/137/144) are all DROPped and recreated to use app.has_share() reading the RLS-protected app.shares table (probe in 0018_probe_owner_or_share.sql:6-18; tasks in 0019_tasks_owner_or_share.sql:8-50 and 0039_tasks_foundation.sql). Every current owner-or-share module policy (calendar 0020, email 0021, chat 0025, briefings 0026, structured-state 0031) calls has_share(), not has_resource_grant(). app.shares HAS RLS (0017_shares.sql:71-72 ENABLE/FORCE, owner-scoped per-op policies 82-110, SECURITY DEFINER has_share() 39-60). So forging/deleting a resource_grants row would NOT widen any user's exposure — the exploit path is closed by construction; resource_grants is now an effectively legacy/dead admin-managed table for access purposes. Residual issue is hygiene (legacy admin-only table lacking defense-in-depth RLS that no longer drives authorization). Note: the finding's "0005 line 20" / "0005_drop_workspaces.sql" references are inexact — 0005 is 0005_admin_audit_events.sql. LOW.
**Suggested fix:** Classify resource_grants as a protected table: ENABLE+FORCE RLS with an admin/owner-scoped policy (or owner-or-grantor policy), revoke broad app_runtime DML in favor of policy-gated writes, and add it to scripts/audit-release-hardening.ts protectedTables so the gate fails if RLS regresses. At minimum add it to the release-hardening audit so the no-RLS state is an explicit, tested decision — or, since it no longer drives authorization, remove it as stale vocabulary.

#### [LOW] Unauthenticated /api/bootstrap/status discloses the total user count of the instance
**File:** packages/settings/src/routes.ts:77
**Invariant violated / concern:** Minimize information disclosure to unauthenticated callers.
**Detail:** GET /api/bootstrap/status has no requireAdmin / resolveAccessContext gate (by design, since it must answer before any user exists) and returns `{ needsBootstrap, userCount }` (routes.ts:77-84). It calls repository.countUsers() which runs the SECURITY DEFINER app.count_all_users() and returns the real total user count (repository.ts:66-71, migration 0045:127-135). A boolean needsBootstrap is sufficient for the bootstrap-detection use case; exposing the exact account count to any anonymous LAN/network caller is unnecessary instance metadata.
**Verification:** Confirmed at LOW. routes.ts:77-84 confirms the handler takes no (request, reply) and calls neither resolveAccessContext nor requireAdmin (unlike every other handler); manifest.ts:53-56 shows it is the ONLY route entry with no permissionId. repository.ts:66-71 runs `SELECT app.count_all_users() AS count`; 0045:127-135 confirms count_all_users() is SECURITY DEFINER returning the true count despite the app role's self-row RLS. apps/api/src/server.ts has no global onRequest/preHandler auth hook and global rate-limit is disabled (:59), so the endpoint is genuinely unauthenticated and unthrottled on 0.0.0.0. The needsBootstrap boolean is the only datum the flow needs; the raw userCount is unnecessary metadata. No private data/secret/token/hash leaked, no RLS bypass on protected data — a minimize-information-disclosure hygiene gap, correctly LOW.
**Suggested fix:** Return only the boolean (needsBootstrap) from the unauthenticated bootstrap endpoint; drop userCount or gate the count behind requireAdmin via the admin users route which already returns it.
