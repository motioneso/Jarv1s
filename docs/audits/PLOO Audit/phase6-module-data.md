#### [MED] jarvis_worker_runtime has DML grants on memory_chunks / memory_file_index but no matching RLS policy (worker chat-embed writes are RLS-denied)
**File:** packages/memory/sql/0040_memory_chat_source.sql:15
**Invariant violated / concern:** Defense-in-depth via RLS + least-privilege roles; runtime paths must function under their actual DB role, and table privileges should not be granted to a role that has no policy permitting row access.
**Detail:** 0040_memory_chat_source.sql:15-16 grants SELECT/INSERT/UPDATE/DELETE on app.memory_chunks and app.memory_file_index to jarvis_worker_runtime so the chat embed-turn job can persist conversation embeddings. But every RLS policy on those two tables is scoped TO jarvis_app_runtime ONLY (memory_chunks: 0030_memory_index.sql:44-63; memory_file_index: 0032_memory_embedding_768.sql:42-61). Both tables are FORCE ROW LEVEL SECURITY (0030:40, 0032:40), and the runtime roles are NOINHERIT/NOBYPASSRLS with no role membership linking worker to app (infra/postgres/bootstrap/0000_roles.sql:45-51, no GRANT jarvis_app_runtime TO jarvis_worker_runtime). Under Postgres FORCE RLS, a role with table grants but no applicable policy is denied ALL rows. The chat embed-turn worker runs as jarvis_worker_runtime: chat persistence enqueues CHAT_EMBED_TURN_QUEUE (packages/chat/src/live/persistence.ts:115); registerBuiltInModuleWorkers wires registerChatJobWorkers (packages/module-registry/src/index.ts:153) onto the worker DB connection (postgres://jarvis_worker_runtime, packages/db/src/urls.ts:27-29, apps/worker/src/worker.ts:14); handleEmbedTurnJob then INSERTs into memory_chunks and memory_file_index (packages/chat/src/jobs.ts:74-93). Those INSERTs are RLS-denied, so chat episodic memory is never persisted. This fails closed (no cross-user leak — the same missing policy also blocks any errant read), so it is a correctness break plus a least-privilege misconfiguration rather than an exploit. Note the asymmetry: chat_memory_facts (0041_memory_facts.sql:28-42) deliberately created its policies with NO role clause (apply to all roles incl. worker), so the worker fact path is correct — memory_chunks/memory_file_index simply did not get the equivalent worker-applicable policy when worker grants were added in 0040.
**Verification:** Confirmed. 0040_memory_chat_source.sql:15-16 grants DML to jarvis_worker_runtime but adds NO policy; all memory_chunks policies are FOR ... TO jarvis_app_runtime only (0030:46,51,56,62) under FORCE RLS (0030:40), all memory_file_index policies likewise TO jarvis_app_runtime only (0032:44,49,54,60) under FORCE RLS (0032:40). Worker connects as jarvis_worker_runtime (apps/worker/src/worker.ts:12-17 -> packages/db/src/urls.ts:27-29); registerChatJobWorkers wired onto that dataContext (module-registry/src/index.ts:152-153); handleEmbedTurnJob INSERTs via upsertFileChunks/upsertFileIndex (chat/src/jobs.ts:74-93). The DataContext layer never SET ROLEs — DataContextRunner only set_config's app.actor_user_id/app.request_id on the worker's own rootDb (db/src/data-context.ts:30-38; jobs/src/pg-boss.ts:91-97). Roles NOINHERIT/NOBYPASSRLS (0000_roles.sql:45-51); only role-membership grant is jarvis_auth_runtime TO jarvis_migration_owner (0000:74) — no worker->app grant. Established pattern confirms omission: chat 0036_chat_worker_runtime_grants.sql:24,38,57, briefings 0015_briefings_module.sql:155,173,186,195, and facts 0041_memory_facts.sql:28-42,45 all co-amend policies to include the worker. Rationale: real fail-closed functional break plus a genuine privilege/policy mismatch (DML granted to a role that can touch zero rows) — MED calibration band.
**Suggested fix:** Add memory_chunks/memory_file_index RLS policies that also apply to jarvis_worker_runtime (either widen the existing FOR ... TO clauses to 'TO jarvis_app_runtime, jarvis_worker_runtime' in a NEW migration, or add worker-scoped owner_user_id = app.current_actor_user_id() policies), mirroring how chat_memory_facts and tasks tables grant both roles. If chat embedding is genuinely intended to be app-runtime-only, instead REVOKE the worker DML grants in 0040 so the grant surface matches the policy surface.

**File:** packages/structured-state/sql/0031_structured_state.sql:60
**Invariant violated / concern:** Shareability classification must be internally consistent: a declared grant level should be honored (or intentionally not) consistently across SELECT/UPDATE/DELETE; silent asymmetry is a latent correctness/authorization gap.
**Detail:** The module manifest declares commitment and entity as shareable with grantLevels ['view','contribute','manage'] (packages/structured-state/src/manifest.ts:28-31). The SELECT policies honor sharing via app.has_share('commitment'/'entity', id, 'view') (0031_structured_state.sql:48-53, 98-103). But the UPDATE policies (0031:60-64 commitments, 110-114 entities) and DELETE policies (0031:66-69, 116-119) are strictly owner_user_id = app.current_actor_user_id() with NO has_share('...','manage') disjunct — unlike tasks, whose UPDATE policy does include the manage-share disjunct (packages/tasks/sql/0019_tasks_owner_or_share.sql:33-50). Net effect: a user granted 'manage' on a commitment/entity can read it but cannot update or delete it. This fails closed (more restrictive than declared), so it is not a privilege-escalation hole — but it is an authorization inconsistency that will silently break the 'manage' feature and contradicts the manifest. The repositories' update()/delete() filter only on id and rely entirely on this RLS USING clause (packages/structured-state/src/commitments-repository.ts:76-86, entities-repository.ts:75-85), so behavior is governed purely by the policy.
**Verification:** Confirmed. manifest.ts:28-31 declares commitment+entity shareable with [view,contribute,manage]. Repos filter only on id (commitments-repository.ts:76-87, entities-repository.ts:75-86). has_share ranks view=1/contribute=2/manage=3 with `>=` (0017_shares.sql:22-60), so a manage-grantee passes the 'view' SELECT disjunct but UPDATE/DELETE USING never consult shares -> writes match zero rows. 0031 is the only SQL in the module; no later migration adds the manage disjunct. Rationale: fails closed (no escalation/leak); genuine authorization-consistency / latent-feature-breakage gap — LOW.
**Suggested fix:** Decide the intended contract and make it consistent: either add the has_share('commitment'/'entity', id, 'manage') disjunct to the UPDATE/DELETE policies (mirroring tasks 0019) in a new migration, or drop 'manage'/'contribute' from the manifest grantLevels until those policies exist, so the declared shareability matches DB enforcement.

#### [LOW] VaultContextRunner builds the per-user vault root by joining actorUserId into a filesystem path with no containment/format validation on the id segment
**File:** packages/vault/src/vault-context.ts:31
**Invariant violated / concern:** All vault I/O is contained to the owner's vault directory; path components derived from identifiers must be validated so a malformed id cannot escape the per-user root.
**Detail:** withVaultContext computes vaultRoot = join(this.vaultsBaseDir, accessContext.actorUserId) and mkdir's it (vault-context.ts:31-32). resolveVaultPath (vault-path.ts:10-23) carefully prevents traversal in the *relative* path argument and asserts the result starts with normalizedRoot — but it validates against the already-computed vaultRoot. The actorUserId segment itself is never validated to be a safe UUID with no path separators or '..'. Today actorUserId is always a server/session-derived UUID (apps/api/src/server.ts:100,199 resolveAccessContext; worker derives it from job.data.actorUserId in packages/jobs/src/pg-boss.ts:100-108), so this is not currently exploitable — it is defense-in-depth. If any future caller ever sourced actorUserId from less-trusted input, a value like '../<otherUser>' would point vaultRoot at another user's vault and every subsequent resolveVaultPath check (anchored to that poisoned root) would pass, yielding cross-user vault content read/write.
**Verification:** Confirmed. vault-context.ts:31 `const vaultRoot = join(this.vaultsBaseDir, accessContext.actorUserId)` then mkdir (32), no format/containment validation on the actorUserId segment before it becomes a path component. vault-path.ts:10-23 validates only the relative path argument and anchors its containment check on `normalizedRoot = resolve(vaultRoot)` (16), accepting whatever vaultRoot it is handed (18) — traversal embedded in actorUserId is never caught. Not exploitable today: actorUserId is constructed only from validated sessions (auth/src/index.ts:228 `session.user.id` after getSession; db/src/auth-session.ts:28 `session.user_id`) and propagated to the worker via the server-populated job payload (jobs/src/pg-boss.ts:106); none flow from raw request input, and better-auth ids are UUIDs. Grep found no UUID format assertion in packages/auth/src or packages/db/src. Rationale: real defense-in-depth gap on a path-derivation routine with no live exploit path (only the absence of an untrusted-input source prevents exploitation today) — correctly LOW.
**Suggested fix:** Validate accessContext.actorUserId against a strict UUID regex (or reject any value containing path separators or '..') inside VaultContextRunner.withVaultContext before join(), and assert the resulting vaultRoot is contained within vaultsBaseDir, so the per-user root cannot be escaped regardless of how actorUserId was sourced.
