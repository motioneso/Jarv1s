#### [MED] Workspace/settings/grant/audit tables have NO RLS yet are granted broad DML to jarvis_app_runtime — protected only by an application-layer requireAdmin check
**File:** infra/postgres/migrations/0004_auth_workspaces_settings.sql:86
**Invariant violated / concern:** Private by default / DB-level defense-in-depth (RLS applies to all actors); no privileged data should be reachable on the basis of an application convention alone.
**Detail:** 0004 GRANTs SELECT/INSERT/UPDATE on app.workspaces, app.workspace_memberships, app.resource_grants, app.instance_settings to jarvis_app_runtime (lines 86-88), 0005:18-20 adds INSERT/SELECT + DELETE on admin_audit_events, workspace_memberships and resource_grants. A tree-wide grep for `ENABLE ROW LEVEL SECURITY`/`CREATE POLICY` over these five tables (infra/postgres/migrations + packages/*/sql) returns NOTHING — they are never RLS-ENABLEd, never FORCEd, and have no policies. Therefore jarvis_app_runtime can SELECT/INSERT/UPDATE/DELETE every row in all of them unconditionally, with no app.current_actor_user_id() scoping. The ONLY thing standing between any authenticated (non-admin) user and these tables is the application-layer `requireAdmin` gate in packages/settings/src/routes.ts:331-344, which checks `user.is_instance_admin` in TypeScript. This is exactly the 'conventions, not DB-level defense' posture the project rejects (CLAUDE.md Hard Invariants + project memory 'Security First'). resource_grants is especially sensitive: it is read by the SECURITY DEFINER authorization function app.has_resource_grant (0002_app_rls.sql:43-63), so its contents are authorization-bearing data with no row-level protection.
**Verification:** Downgraded HIGH → MED. Confirmed: 0004:86-88 grants SELECT/INSERT/UPDATE on the four tables; 0005:18 grants SELECT,INSERT on admin_audit_events and :20 grants DELETE on workspace_memberships+resource_grants to jarvis_app_runtime. Per-table grep for ENABLE/FORCE ROW LEVEL SECURITY and CREATE POLICY across infra/postgres + packages/*/sql returns NOTHING for any of the five tables. jarvis_app_runtime is NOBYPASSRLS NOINHERIT (bootstrap/0000_roles.sql:35-43) but with no policies and FORCE unset the GRANTs alone give unconditional all-row access. SettingsRepository (packages/settings/src/repository.ts:64) runs every query on these tables via the jarvis_app_runtime appDb. The sole control is the TS requireAdmin gate (routes.ts:331-344); /api/me (routes.ts:86-91) reads workspaces/memberships but is self-scoped via listMembershipsForUser/listWorkspacesForUser(actorUserId). resource_grants remains authorization-bearing: app.has_resource_grant (0002:43-63, SECURITY DEFINER) feeds the tasks UPDATE/DELETE policies (packages/tasks/sql/0003_tasks_module.sql:137,144). Rationale: the finding is real, but HIGH overstates present exploitability — every route touching these tables is behind requireAdmin and the one ungated route is self-scoped, so a non-admin has no demonstrated path today. This is a latent defense-in-depth gap (missing-by-default protection not yet proven exploitable) = MED.
**Suggested fix:** Add a migration that ENABLE+FORCE ROW LEVEL SECURITY on app.workspaces, app.workspace_memberships, app.resource_grants, app.instance_settings, app.admin_audit_events and attach explicit policies. Since these are instance/admin-management surfaces, the correct policy is likely a dedicated admin predicate (the planned app.current_actor_is_admin() helper) rather than owner-scoping, plus an append-only / no-self-tamper rule for admin_audit_events. Until that helper lands, at minimum gate writes on `app.current_actor_user_id() IS NOT NULL` and confirm requireAdmin cannot be bypassed by any non-settings route that holds the same appDb.

#### [MED] SettingsRepository runs all queries on a raw Kysely<JarvisDatabase> with no DataContext / no actor GUC set, bypassing the DataContextDb-only invariant
**File:** packages/settings/src/repository.ts:64
**Invariant violated / concern:** DataContextDb only — repositories accept only a branded DataContextDb handle, never a root Kysely instance; the actor GUC (app.actor_user_id) must be set transaction-locally before any RLS-governed query.
**Detail:** SettingsRepository's constructor takes `private readonly db: Kysely<JarvisDatabase>` (line 64) and SettingsDb is `Kysely | Transaction` (line 16). Every method (countUsers, getUserById:74, listUsers:77, listWorkspaces:86, listResourceGrants:262, listInstanceSettings:348, listAdminAuditEvents:388, plus the write paths) executes directly against this raw handle. It is constructed in packages/settings/src/routes.ts:75 from `dependencies.appDb` (the app_runtime pool). None of these calls go through DataContextRunner.withDataContext, so `app.actor_user_id` is NEVER set on the connection — unlike every other repository in the codebase, which receives a branded DataContextDb (assertDataContextDb). Today this 'works' only because the tables it touches happen to have no RLS (see related MED) and because app.users has a deliberate `users_app_runtime_select USING (true)` policy (0045_auth_secret_rls.sql:83-87). The moment RLS is added to the workspace/settings tables (the correct fix for the related finding), these queries would silently return zero rows because the actor GUC is unset and app.current_actor_user_id() returns NULL. This repository is a standing exception to the project's central data-access invariant.
**Verification:** Confirmed at MED. repository.ts:16 `type SettingsDb = Kysely<JarvisDatabase> | Transaction<JarvisDatabase>;` and :64 constructor takes a raw root Kysely. Every read and every write/transaction (createWorkspace:138, upsertWorkspaceMembership:184, upsertResourceGrant:274) runs on `this.db` with no `set_config('app.actor_user_id', ...)`; grep of packages/settings/src shows zero withDataContext / DataContextRunner / assertDataContextDb / set_config. Wiring: routes.ts:75 `new SettingsRepository(dependencies.appDb)`; apps/api/src/server.ts:32-37 builds appDb from getJarvisDatabaseUrls().app (jarvis_app_runtime pool) and passes the SAME instance to both `new DataContextRunner(appDb)` (server.ts:53) and the settings module (server.ts:99). Contrast packages/db/src/sharing/shares-repository.ts:24-25 and packages/db/src/probes/rls-probe-repository.ts:6-7, which take DataContextDb and call assertDataContextDb. "Works today" verified: the five workspace/settings/grant/audit tables have NO ENABLE RLS (only app.users at 0045:51 and rls_probe_items), and app.users carries users_app_runtime_select USING (true) (0045:82-87). Conditional break verified: app.current_actor_user_id() returns NULL when the GUC is unset (0002:9-13), so any future owner-scoped policy on these tables would silently match zero rows. Runtime roles are NOBYPASSRLS (bootstrap/0000_roles.sql:35,43,51,59), so no admin-bypass invariant is broken. Rationale: a real defense-in-depth / invariant-consistency gap and latent foot-gun, but not currently exploitable (no secret escape, no RLS bypassed) and it directly violates the DataContextDb-only hard invariant on a real runtime path = MED.
**Suggested fix:** Route SettingsRepository through the DataContext path (accept DataContextDb / run inside withDataContext so app.actor_user_id is set), or, if genuine cross-user admin reads are required, make that explicit via a SECURITY DEFINER admin function owned by an admin role (mirroring app.count_all_users / app.list_connector_account_safe_metadata) rather than a raw unscoped Kysely handle. Either way, eliminate the raw `Kysely<JarvisDatabase>` constructor parameter so the branding invariant holds uniformly.

#### [MED] users_app_runtime_update policy is row-scoped but not column-scoped: an app_runtime self-row UPDATE could flip is_instance_admin (privilege self-escalation, defense-in-depth gap)
**File:** infra/postgres/migrations/0045_auth_secret_rls.sql:96
**Invariant violated / concern:** No admin private-data bypass / admin power is configuration power gated at the DB layer; a user must not be able to grant themselves instance-admin.
**Detail:** users_app_runtime_update (0045_auth_secret_rls.sql:96-102) uses `USING (id = app.current_actor_user_id()) WITH CHECK (id = app.current_actor_user_id())`. RLS restricts WHICH ROW may be updated, never WHICH COLUMNS. The table GRANT (0004:79-80) is a column-unqualified `UPDATE ON app.users`. So at the DB layer, jarvis_app_runtime — when its actor GUC is set to the user's own id — may UPDATE its own row including `is_instance_admin = true`. No DB-level constraint, trigger, or column GRANT prevents self-escalation. I found no application route in the tree that performs an arbitrary user-column UPDATE under app_runtime (SettingsRepository has no user-update method; better-auth profile updates run as jarvis_auth_runtime), so this is not a confirmed live exploit path today — but it is a latent defense-in-depth gap directly contradicting the no-admin-bypass invariant, and would become exploitable the instant any app_runtime code path updates user rows.
**Verification:** Confirmed at MED. 0045:96-102 — `CREATE POLICY users_app_runtime_update ON app.users FOR UPDATE TO jarvis_app_runtime USING (id = app.current_actor_user_id()) WITH CHECK (id = app.current_actor_user_id());` — row-scoped only, no column predicate. 0004:79-80 — `GRANT SELECT, INSERT, UPDATE ON app.users TO jarvis_app_runtime;` — column-unqualified, so is_instance_admin is writable. 0001_app_schema.sql:22 — `is_instance_admin boolean NOT NULL DEFAULT false` is a real privilege flag gating admin routes (connectors/src/routes.ts:267, settings/src/routes.ts:339). No DB-level trigger/CHECK/column-grant guards it. The one existing app_runtime user-row UPDATE — packages/auth/src/index.ts:252-263 — explicitly sets the actor GUC to the user's own id (252) and writes `is_instance_admin: isFirstUser`, proving the self-row-UPDATE-touching-this-column shape is already live; but there the value is a server-computed advisory-locked count, not attacker input, and better-auth declares the field `input: false` (index.ts:151), so this specific path is safe. SettingsRepository exposes only getUserById/listUsers/countUsers (no user write) and there is no better-auth updateUser/changePassword call in app code. Rationale: the DB does not column-scope user updates, so under the no-admin-bypass invariant the DB does not prevent self-escalation; rated MED rather than CRIT/HIGH because there is no live exploit path (the sole writing route uses a server-computed value under an advisory lock with input:false). This is a defense-in-depth gap not yet proven exploitable = MED.
**Suggested fix:** Restrict the app_runtime UPDATE to a safe column set via a column-level GRANT (GRANT UPDATE (name, image, email_verified, updated_at) ON app.users TO jarvis_app_runtime) and/or a BEFORE UPDATE trigger that rejects changes to is_instance_admin unless performed by an admin-privileged role. is_instance_admin should only be mutable through an explicit, audited admin path.

#### [LOW] memory_facts RLS policies omit the TO clause, applying to PUBLIC instead of the named runtime roles (consistency deviation from every other module)
**File:** packages/memory/sql/0041_memory_facts.sql:1
**Invariant violated / concern:** Module RLS policies are scoped to the explicit runtime roles (TO jarvis_app_runtime, jarvis_worker_runtime), matching the project-wide pattern.
**Detail:** Every other module's policies attach `TO jarvis_app_runtime[, jarvis_worker_runtime]` (e.g. tasks 0019:15, calendar, chat, ai). The memory_facts policies are FORCE-RLS with correct owner-scoped `owner_user_id = app.current_actor_user_id()` USING/WITH CHECK predicates but carry no TO clause, so they apply to PUBLIC. Functionally this is still fail-closed (the owner predicate holds for whatever role connects, and only the two runtime roles hold the table GRANT), so it is not a confirmed vulnerability — but it is an inconsistency that weakens the per-role audit boundary and could surprise a future reader who assumes role-scoped policies everywhere.
**Verification:** Confirmed at LOW. 0041_memory_facts.sql:28-42 — all four policies (chat_memory_facts_select/insert/update/delete) lack a `TO <role>` clause, so they apply to PUBLIC. Contrast the canonical pattern: tasks/sql/0019_tasks_owner_or_share.sql:15 (`TO jarvis_app_runtime, jarvis_worker_runtime`), memory/sql/0030_memory_index.sql:46/51/55/62, structured-state/sql/0031_structured_state.sql:49/57/62/68. An exhaustive scan of all 114 CREATE POLICY statements under packages/*/sql shows exactly 8 lacking a `TO jarvis` clause: the 4 here plus the 4 in packages/chat/sql/0042_chat_memory_settings.sql:16-29; the other 106 carry the explicit scope. Fail-closed proof: (a) the predicate uses app.current_actor_user_id() which RETURNS NULL when the GUC is unset (0002:9-13), so the owner comparison yields no rows absent a set actor; (b) the only table GRANTs are to the two runtime roles (0041:44-45) with FORCE RLS (0041:25-26); (c) there are zero `TO PUBLIC` grants in the tree and EXECUTE on current_actor_user_id() is revoked from PUBLIC, granted only to runtime roles (0002:85-93). Rationale: a defense-in-depth / per-role-audit-boundary consistency gap with no reachable PUBLIC subject and no exploit path = LOW. Note: the same omission also affects packages/chat/sql/0042_chat_memory_settings.sql:16-29 (separate finding below).
**Suggested fix:** In a NEW migration (never edit applied 0041), re-create the memory_facts policies with explicit `TO jarvis_app_runtime, jarvis_worker_runtime` to match the project-wide convention.
