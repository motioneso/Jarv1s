### Bug 1 (real plan gap ‚Äî not in the plan at all)

`apps/api/src/server.ts` calls `registerBuiltInApiRoutes(server, {...})`, whose dependencies type
is **`BuiltInRouteDependencies`**, defined in **`packages/module-registry/src/index.ts`** (NOT
`SettingsRoutesDependencies` directly ‚Äî `registerBuiltInApiRoutes` is a wrapper in module-registry
that forwards a subset of its deps into `registerSettingsRoutes`). The plan's Step 8 never
mentioned this intermediate type, so passing `moduleDistribution` in server.ts's dependencies
object fails: `error TS2353: Object literal may only specify known properties, and
'moduleDistribution' does not exist in type 'BuiltInRouteDependencies'.`

Fix, mirroring the existing `externalModules` field exactly (same file):

1. `packages/module-registry/src/index.ts` **~L176-177** already imports from `@jarv1s/settings`:
   ```ts
   type ExternalModulesDependencies
   } from "@jarv1s/settings";
   ```
   Add `ModuleDistributionDependencies` to that same import (it's exported from
   `packages/settings/src/routes.ts` and flows through `@jarv1s/settings`'s index via
   `export * from "./routes.js"` ‚Äî confirmed already in place from this relay's Step 6b work).

2. `BuiltInRouteDependencies` interface, right after the existing
   `readonly externalModules?: ExternalModulesDependencies;` field (**~L440**), add:
   ```ts
   readonly moduleDistribution?: ModuleDistributionDependencies;
   ```

3. In the `BUILT_IN_MODULES` array's `settingsModuleManifest` entry's `registerRoutes` closure
   (**~L892**), right after the existing line
   `externalModules: deps.externalModules, // #917: thread the boot snapshot to settings routes`,
   add:
   ```ts
   moduleDistribution: deps.moduleDistribution,
   ```

Re-run `pnpm --filter @jarv1s/module-registry typecheck` and
`pnpm --filter @jarv1s/api typecheck` after this fix to confirm it resolves cleanly (module-registry
itself must still typecheck ‚Äî it's importing a new type from settings).
