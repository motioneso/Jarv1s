### systemd-creds + codex alternative — feasibility verdict (informational, Ben asked; NOT chosen)

`systemd-creds` is present (systemd 255, TPM2-backed) but it's a host/systemd-unit decryption
mechanism (`LoadCredentialEncrypted=` inside a systemd unit) — this stack runs under docker-compose,
not systemd units, so using it would mean bolting a new systemd-unit layer around cli-runner just to
decrypt. Codex's own credential lives at a separate on-disk path (`CODEX_HOME=/data/cli-auth`,
`.codex/config.toml`), not `provider-token-store.ts`'s claude-scoped file, and has **no static-token
capture** — login-adapters.ts's codex adapter is `mode:"poll"` device-auth only. Pre-seeding it would
require either a live interactive `codex login --device-auth` approval per provisioning event, or
copying codex's own auth file (exactly the "no auth-file copying" constraint rules out). **Verdict:
Claude/Option A is strictly more feasible** — zero interactive login at provisioning time (decrypt +
inject a pre-minted static token); codex would need a live human approval every single run. This
matches what's already approved below — do not revisit unless Ben asks again.
