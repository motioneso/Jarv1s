## Scope — do exactly this, nothing more

Replace the explicit `throw` stub at **`tests/uat/seed/levels.ts:61-66`** (currently:
`throw new Error("seedLevel: multi-user is deferred to #1030 …")`) with a real `multi-user` seeding
path:

- **≥2 distinct owners**, each with realistic per-owner lived-in data, seeded through the **same
  real-repo + `app_runtime` + per-owner `AccessContext` path** that `admin+data` already uses
  (`createAppRuntimeRunner` + the `ADMIN_DATA_CHUNKS` seeders). The `admin+data` block already
  seeds owner #1 (the admin); `multi-user` extends it with a **second, non-admin owner** who has
  their own data.
- **No RLS carve-out. No `BYPASSRLS`.** The second owner's data MUST be written under **that
  owner's own `AccessContext` (`actorUserId` = owner2's id)** — never under the admin's context —
  so RLS scopes it naturally. This is the security crux: a seed that writes owner2's rows under
  owner1's context is exactly the cross-user leak this level exists to catch. (CLAUDE.md Hard
  Invariant: AccessContext = `{actorUserId, requestId}` only; admin power is config power, RLS
  applies to all actors.)
- **Cross-user share/RLS fixtures:** seed at least one explicit cross-owner *share* (owner1 shares
  a resource with owner2, per the module's real share/grant API — not a raw INSERT) so the UI can
  exercise the owner-or-share privacy path, plus data that stays private to each owner so the
  isolation boundary is visible. Use the real grant/share repository method; do not hand-write a
  grant row.
- **Deterministic:** fixed injected base timestamp (reuse the existing `timestamps.ts` injection
  the other seeders use — do not call `Date.now()` / `new Date()`). Same seed → same rows.
- Create the second owner the same way owner1 is created (mirror `seedSoloAdmin` in `admin.ts` for
  the account/user creation, but as a **non-admin** owner). If a helper needs generalizing, keep
  the change minimal and preserve `seedSoloAdmin`'s existing behavior for the admin path.

## Exit criteria

- `seedLevel({ level: "multi-user", … })` runs to completion (no `throw`), seeding ≥2 owners with
  per-owner data + at least one cross-owner share, under each owner's own `AccessContext`.
- A test proves **isolation**: owner2's private rows are NOT visible under owner1's `AccessContext`
  and vice-versa, and the shared resource IS visible to the share recipient (owner-or-share path).
  This is the security-defining test — make it explicit and adversarial, not incidental.
- Reseed is idempotent (no duplicate rows); runner pool closed on both success and error paths.
- Full local gate green: `pnpm verify:foundation` (or record exact commands + exit codes if CI is
  the gate). File-size + format:check + typecheck + lint must pass. Run the UAT seed suite.
