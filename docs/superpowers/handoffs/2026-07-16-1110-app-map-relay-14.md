## What changed since -13

Ran `pnpm test:uat -- app-map-grounding` (first execution of the rewrite). **2/3 passed.**
Test1 ("declared prerequisite surfaces the News no-json-model error") failed — but **not** a
selector miss. Root-caused a real seed-logic bug:

**Root cause:** `AiRepository.resolveModelForService` (`packages/ai/src/repository.ts:1172`)
falls through to `resolveModelForCapability`'s **implicit-default-provider** path whenever
`module.news` has no explicit binding (`resolveDefaultProviderId`, line 812: "with no flag,
exactly one active admin-owned provider is the implicit default"). `seedAiProviderChunk`
(`tests/uat/seed/chunks/ai.ts`) creates the fake provider+model **unconditionally**, gating
only the `setServiceBinding` call on `options.bindNews`. So with `bindNews:false` the fake
provider is still the sole active admin-owned provider → becomes the implicit default →
`hasJsonModel()` resolves **true** via fallback anyway → test1's "example.com" input skips the
`no_json_model` gate and instead fails on a real network-reachability error ("We couldn't reach
that site" / `discovery_unavailable`). Confirmed in the failure's `error-context.md` snapshot
("AI model ready" badge was showing — proves the fallback fired).

**Fix (verified safe, NOT yet applied):** in `tests/uat/seed/chunks/ai.ts`, early-return
(skip creating provider+model entirely) when `!options.bindNews`, so `resolveDefaultProviderId`
sees zero providers → `hasJsonModel()` genuinely false. Checked no other chunk depends on this
provider: grepped `tests/uat/seed/chunks/{news,sports,onboarding}.ts` for
`modelId|providerConfigId|model_id|provider` — no hits. `ai.test.ts` and `news.test.ts` both
call `seedAiProviderChunk(runner, userId)` with the **default** `bindNews:true` — unaffected by
an early-return gated on `false`.
