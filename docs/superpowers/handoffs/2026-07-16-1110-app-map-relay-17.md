## Root cause — found via static analysis, NOT yet empirically confirmed by a live repro

Invoked `superpowers:systematic-debugging` (Phase 1 in progress). Evidence chain:

1. This PR's diff to `packages/module-registry/src/index.ts` adds `appMapService =
   createAppMapReadService({ artifact: loadAppMap(APP_MAP_ARTIFACT_PATH), ... })` inside
   `registerBuiltInApiRoutes()` — called synchronously during API **boot**, not lazily.
2. `APP_MAP_ARTIFACT_PATH = <workspaceRoot>/dist/app-map.json` (computed by walking up to
   `pnpm-workspace.yaml`).
3. `packages/settings/src/app-map.ts:27` `loadAppMap()` does a bare `readFileSync(path,
   "utf8")` — **no try/catch, no fallback.** Throws `ENOENT` if the file is missing → crashes
   the whole API process at startup.
4. `dist/` is gitignored (`.gitignore:5`), `dist/app-map.json` is untracked — confirmed via
   `git check-ignore -v dist/app-map.json`. Nothing writes it except `scripts/build-app-map.ts`
   (script name: `build:app-map`).
5. `apps/api/package.json`: `"dev": "pnpm --dir ../.. build:app-map && tsx watch
   src/server.ts"` — runs the generator first. **But `"start": "tsx src/server.ts"` does
   NOT.**
6. `scripts/build-app.ts:79` (`buildTarget("api")`, used by the production Docker image build)
   also runs `execFileSync("pnpm", ["build:app-map"], ...)` first — **this is why "Prod
   compose deployment smoke" passes** (image build generates the artifact before packaging).
7. `infra/docker-compose.yml`'s `api` service command is `... && pnpm start:api` (root
   `start:api` → `apps/api`'s `"start"`, NOT `"dev"`) — **never generates the artifact.** On a
   fresh CI checkout (dist/ doesn't exist, gitignored), `readFileSync` throws ENOENT at boot →
   crash-loop under `restart: unless-stopped` → healthcheck never passes → smoke test times
   out.

**Conclusion: root cause is a missing `build:app-map` step in the dev-compose `api` service's
command, not a code logic bug per se.** High confidence but NOT YET empirically reproduced
live — my own worktree already has a stale `dist/app-map.json` on disk (from earlier
`pnpm dev`/test activity this relay chain), which would mask the bug if I just ran `docker
compose -f infra/docker-compose.yml up` right now (the bind-mount `..:/workspace` would carry
my host's `dist/` into the container). **Note: `dist/` is permission-denied for Read/ls/find
listing by this session's tool settings — `test -f dist/app-map.json` works, but `ls`/`find
-maxdepth 2`/Read on files under `dist/` are denied.** Don't fight that; work around it (see
next steps).
