## Issue #539 Direction

Spec direction:

- Build on #525 cross-tool reasoning, #530 passive context retrieval, #532 confidence-aware memory,
  #537 commitment evidence, and #538 person context citation tokens.
- Define user-visible answer provenance for Jarvis answers, not a generic citation engine.
- Answers should be able to show compact source-backed support: source labels, snippets when safe,
  source kind, confidence/provenance, and links/deep links through source-owned verifiers.
- Source cards/citations must be owner-scoped and bounded. Do not expose raw source refs, connector
  payloads, prompt text, secrets, full email/note/chat bodies, or raw memory records.
- Hidden context blocks from #525/#530 are not themselves provenance. #539 should define how
  source-backed items are carried from source-owned reads into answer metadata and UI.
- Use source-owned verifier/dereference paths. The central answer/provenance layer must not query
  source-owned tables directly.
- Distinguish confirmed sources, inferred memory, pending candidates, and ambiguous people. Pending
  #537 commitments and ambiguous #538 people should be labeled as such.
- Tie action audit details to #540 and data freshness/staleness wording to #541. Do not build
  freshness badges here beyond basic timestamps needed for provenance.
- Do not build connector sync, a document browser, a global search UI, or a general knowledge-base
  export feature.

## Issue #540 Direction

Spec direction:

- Build on #527 usefulness feedback, #531 proactive cards, #534 action permission tiers, #535
  goals, #536 scheduled briefings, #537 commitment candidate events, and #538 person context events.
- Define a safe automation audit log for Jarvis actions and action-like decisions.
- Capture metadata-only records for assistant tool proposals, approvals, denials, trusted auto-runs,
  destructive confirmations, scheduled/proactive suggestion creation, and candidate/status
  transitions that can later explain why Jarvis suggested or did something.
- This is not a raw app log viewer. It should be owner-facing, bounded, private, and useful for
  answering "what did Jarvis do, suggest, or decide not to do?"
- Audit rows must not store raw action inputs, prompts, source bodies, connector payloads, secrets,
  tokens, full summaries, or raw tool outputs. Use ids/hashes, source labels, tool names, risk,
  action family, permission tier, status, timestamps, and verifier-backed references.
- Preserve #534's hard floor: destructive/external-send actions always confirm. Audit logging must
  not become a second executor or permission path.
- Scheduled/proactive surfaces may record suggestions and user decisions, but they must not execute
  actions outside the gateway.
- Tie answer provenance/citations to #539 and freshness labels to #541. Do not build provenance UI
  or freshness warnings here.
