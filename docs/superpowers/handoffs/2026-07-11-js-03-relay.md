## The RED finding (verbatim-critical)

**B1 BLOCKING — truth-guard vacuous pass:** the guard only verifies claims the AI self-reports
in `materialClaims`. An AI response can put fabricated facts in `proposedMarkdown` while
returning `materialClaims: []` — `verifyClaims([])` returns `ok:true`, so the unverified
markdown persists as a draft with `evidence: []` (resume.ts:340→357) and is then approvable to
active (resume.ts:393). Prompt instructions are NOT a security boundary. Required: derive
material content from the markdown itself; refuse to persist/approve markdown containing factual
assertions not backed by verified evidence. **Fail CLOSED.** Required tests: BOTH a unit and an
integration case for "AI under-declares materialClaims (e.g. []) while fabricating in
proposedMarkdown" — assert it CANNOT be persisted-as-approvable / cannot reach active.

min-length — trivial token like `"40%"` marks a claim sourced (unit test at lines ~168–176
codifies the weak behavior). Add min-length requirement + update that test.

## The fix design (COMPLETE — R11 read all target files; build directly from this)

### 1. truth-guard.ts (`external-modules/job-search/src/domain/truth-guard.ts`, 217 lines)

- Add `CLAIM_QUOTE_MIN_CHARS = 12`. In `verifyClaims`, a quoted claim with
  `claim.quote.trim().length < 12` → unsupported. Deliberately NOT added to `CRITIQUE_SCHEMA` /
  `parseCritique` — schema failure hits the seam-error path; we want the recoverable "question"
  path, so enforce in `verifyClaims` only.
- Add `extractMaterialSpans(markdown)`: split into segments = lines with markdown syntax
  stripped (`#`, `-`, `*`, `>`, ordered-list prefixes `1.`, emphasis markers). Material tokens =
  (a) any token containing a digit (punctuation-stripped); (b) capitalized words (≥2 chars)
  mid-segment; for a segment-INITIAL capitalized run, drop only the FIRST word (sentence-case
  allowance) — remaining run words are material.
- Add `verifyMarkdownCoverage({ markdown, sources, confirmedTexts })`: every material token must
  be a case-insensitive substring of the allowed corpus = source revision contents +
  user-CONFIRMED claim texts. **CRITICAL bypass avoidance: the corpus must NOT include
  AI-declared claim texts of sourced claims** — the AI could attach a legit quote to a claim
  whose `text` contains fabricated tokens, whitelisting them. Only user confirmations
  (first-write-wins records) and stored sources vouch.
- Export new symbols; re-export from `src/domain/index.ts`.

### 2. confirmations.ts (`external-modules/job-search/src/domain/confirmations.ts`, 69 lines)

- Add `listConfirmations(kv)` returning FULL `ConfirmationRecord`s (they already store
  `claimKind` + `claimText`): list keys by prefix `confirmation/` in NS.resume, readRecord each.
  Existing `listConfirmationIds` stays.

### 3. resume.ts (`external-modules/job-search/src/worker/handlers/resume.ts`, 404 lines)

- In `critiqueSave`, after the existing `verifyClaims` verdict: call `verifyMarkdownCoverage`
  with `critique.proposedMarkdown`, the same `sources` array, and
  `confirmedTexts = (await listConfirmations(ports.kv)).map(r => r.claimText)`. If EITHER
  verdict fails → return `status: "question"` carrying `unsupportedClaims` and/or the unverified
  spans; **nothing persisted**. Blocking at persist kills the approve path too: the draft
  revision never exists, so `approveResume` rejects `missing_revision` (resume.ts:393 path).
- Update `CRITIQUE_INSTRUCTIONS`: state quote length guidance (≥12 chars, substantive span) and
  that every factual token in proposedMarkdown must come from the provided sources or confirmed
  claims — but comment that the prompt is guidance, the guard is the boundary.
- Rich why-comments citing the QA finding + both verdict comment ids (Ben wants generous
  why-comments).
