## Issue #535 Direction

Spec direction:

- Build on #526 unified priority model, #527 usefulness feedback, #528/#532 memory graph/confidence,
  #533 memory dashboard, and #534 action permission tiers.
- Define long-running Jarvis goals as user-owned, durable, reviewable objectives that can persist
  across days or weeks.
- Goals should track status, priority, target cadence/checkpoints, source links, recent progress,
  blockers, and next suggested action.
- Distinguish goals from tasks. Tasks remain the concrete action substrate; goals organize and
  contextualize tasks, memory, and briefings.
- Do not invent a full workflow engine.
- Do not allow goals to execute actions directly. Goal-driven suggestions must route through #534.
- Keep private by default and owner-only in V1. No admin private-data bypass.
- Use memory records as context and evidence, not as the goal store.
- Tie future commitment extraction to #537 and person/contact enrichment to #538.

## Issue #536 Direction

Spec direction:

- Build on existing briefings work, #526 priority model, #531 restrained proactive monitoring, #534
  action permission tiers, and #535 long-running goals.
- Define scheduled recurring Jarvis briefings as explicit user-configured briefing schedules, not
  proactive cards and not arbitrary background chat turns.
- Include schedule/cadence, timezone, briefing type, enabled state, source selection, quiet-hours
  behavior, and delivery/listing semantics.
- Preserve metadata-only job payloads; scheduled jobs carry actor/schedule ids and command metadata,
  never private briefing content or prompts.
- Source permissions and source behavior settings must be separate from schedule enablement.
- Briefings may suggest actions, but execution must route through #534.
- Do not build push/mobile notification delivery unless an existing notification path already covers
  it with no new product surface.
- Tie audit-log detail to #540 and freshness labels to #541.
