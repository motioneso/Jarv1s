## Issue #531 Notes

Issue body summary:

- Jarvis needs to watch tasks, calendar, Gmail, and notes for important changes.
- It should surface only high-signal items.
- Acceptance sketch: monitorable source changes, high-signal thresholds, anti-spam limits, delivery
  surfaces, quiet hours, user-visible controls, and per-source permissions.

Spec direction:

- Keep V1 restrained. It should produce candidate proactive cards/suggestions, not execute actions.
- Reuse #526 for priority scoring and #527 for usefulness feedback, especially `proactive_card`.
- Define a module-owned provider/registry shape for source change monitors; do not let a monitoring
  service query module-owned tables directly.
- Define anti-spam limits: per-source caps, quiet hours, cooldowns, duplicate suppression, and
  dismissal/feedback suppression.
- Define delivery surfaces: today view/proactive surface first; avoid push notifications unless a
  later permission spec enables them.
- Keep source permissions private-by-default and owner-scoped.
- Defer scheduled recurring briefings to #536, permission tiers to #534, audit log details to #540,
  and data freshness labels to #541.

## Issue #532 Notes

Issue body summary:

- Memory needs to distinguish facts, preferences, guesses, stale records, and outdated information.
- Jarvis answers should phrase memory-backed claims based on confidence/state.
- Acceptance sketch: memory state/type fields, answer phrasing by confidence, correction and
  expiration flows, confidence/state shown in the future user-editable memory surface.

Spec direction:

- Align with #529 memory candidates and existing memory specs; do not invent a second memory store.
- Keep V1 metadata small: record kind, confidence, state, stale/superseded markers, and provenance
  needed for safer answer phrasing.
- Define how answers should phrase high-confidence facts versus preferences, inferences, guesses,
  stale records, rejected/superseded records, and conflicting records.
- Define correction/expiration flows at the API/contract level, while leaving the full dashboard UI
  to #533.
- Tie provenance/source-backed answer needs to #539 and freshness display to #541.
- Preserve owner-only RLS and no admin private-data bypass.
