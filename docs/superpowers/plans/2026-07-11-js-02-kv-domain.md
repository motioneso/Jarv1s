## Global constraints

- **No migration, no core table, no `@jarv1s/db`, no raw fs** anywhere in this slice.
- Namespaces exactly as the shipped manifest: `job-search.{onboarding,profile,resume,monitors,opportunities,runs,feed}`, all user-scope.
- Key ABI (module-design table — JS-03/05/06 root on it): onboarding `state`; profile `active`,
  `revision/<id>`; resume `active`, `revision/0`, `revision/<id>`; monitors `monitor/<id>`,
  `cursor/<id>`; opportunities `job/<identity-hash>`, `tombstone/<identity-hash>`; runs
  `run/<monitorId>/<runId>`, `monitor/<id>/latest`; feed `active`.
- Keys carry ids/hashes only — ids validated `/^[A-Za-z0-9_-]{1,64}$/`, hashes 32-hex. Never
  prose/URLs/titles in keys.
- Every value: JSON object with `schemaVersion: 1`; serialized size `<= 65_535` bytes (below
  65,536; DB check is `<= 65536` — domain cap is strictly tighter).
- Fixed limits (`limits.ts`, exact): `KV_VALUE_MAX_BYTES = 65_535`,
  `RESUME_INPUT_MAX_BYTES = 49_152` (48 KB), `DESCRIPTION_MAX_BYTES = 16_384` (16 KB),
  `OPPORTUNITY_TARGET = 500`, `PASSED_STALE_EVICT_DAYS = 30`, `TOMBSTONE_TTL_DAYS = 60`,
  `RUN_RETENTION_MAX = 50`, `RUN_RETENTION_DAYS = 14`.
- No ambient time: every time-dependent function takes `now: Date` (repo rule, also testability).
- Write ordering: canonical record first, derived pointer/index second; derived writes idempotently
  rebuildable. No cross-key transaction assumed.
- Typed scrubbed errors (`JobSearchKvError` with `code`; messages NEVER embed record content):
  `invalid_schema_version | invalid_record | oversize_value | resume_input_too_large |
missing_active_pointer | missing_revision | missing_record | immutable_revision_conflict |
corrupt_index`. Resume rejection copy (exact export):
  `RESUME_TOO_LARGE_MESSAGE = "Resume text is over the 48 KB limit (49,152 bytes of UTF-8). Trim it and paste it again."`
- Tests live in top-level `tests/unit/` + `tests/integration/`, named
  `external-module-job-search-kv-*.test.ts` (JS-01 convention).
- `git add` explicit paths only. Commit trailer `Co-Authored-By: Claude Fable 5 <noreply@anthropic.com>`.
- Prettier-format every new md/ts before commit (`pnpm prettier --write <paths>`).

## Behavior rules (implement exactly)

- **Immutability:** writing an existing profile/resume revision with DIFFERENT content →
  `immutable_revision_conflict`; byte-identical rewrite → idempotent no-op. `revision/0` original
  resume is retained unchanged forever; `saveResumeRevision` refuses `revisionId === "0"`.
- **48 KB gate:** `Buffer.byteLength(content, "utf8") > 49_152` → `resume_input_too_large` with
  `RESUME_TOO_LARGE_MESSAGE`, checked BEFORE any write, for BOTH original and normalized content.
- **Approval:** `approveProfile`/`approveResume` verify the revision exists (`missing_revision`),
  then write only the `active` pointer. `getActive*`: no pointer → `null`; pointer present but
  revision record gone → `missing_active_pointer` (fail-closed, never silently null).
- **Description cap:** `upsertOpportunity` truncates `posting.description` to ≤ 16_384 bytes on a
  UTF-8 boundary and sets `descriptionTruncated: true`.
- **Idempotent upsert:** same identity + same `contentHash` retried → same record with refreshed
  `lastSeenAt` (and `firstSeenAt`/`status`/`statusAt` preserved); changed `contentHash` updates
  posting + `contentHash` + `lastSeenAt`, preserves user status. An ACTIVE (unexpired at `now`)
  tombstone suppresses re-ingestion: upsert returns `{ suppressed: true }` and writes nothing; an
  expired tombstone is deleted and ingestion proceeds.
- **Retention pass order:** (1) evict `passed`/`stale` with `statusAt` older than 30 days;
  (2) while non-protected count still > 500, evict oldest `lastSeenAt` first among
  `new`/`passed`/`stale`; NEVER evict `saved`/`active` — count them as `protectedOverflow` when
  protected records alone exceed 500 (allowed overflow, reported not refused); (3) each eviction
  writes the tombstone BEFORE deleting the job key (interrupted pass = re-evictable, idempotent);
  (4) delete tombstones with `expiresAt <= now`; (5) prune runs per monitor: keep newest ≤ 50 AND
  newer than 14 days (intersection — "whichever is smaller"), refresh `monitor/<id>/latest` if its
  run was pruned (keep summary — it summarizes, not references); (6) `rebuildFeed`; (7) return
  `RetentionReport`.
- **Feed:** derived only, rebuilt from `job/*` canonical records sorted `lastSeenAt` desc.
  `readFeed` on unparseable/invalid-schema index throws `corrupt_index`; `readFeedOrRebuild`
  catches exactly that and rebuilds. A failed/interrupted feed write never loses a posting.
