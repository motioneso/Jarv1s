# Plan — #650 connector sync wedge (stranded exclusive singleton)

## Root cause (confirmed by reading pg-boss@12.18.2 source)

`resolvePgBossConstructorOptions` (`packages/jobs/src/pg-boss.ts`) hard-defaults
`supervise: false`. `apps/worker/src/worker.ts` overrides only `schedule: true`
(`WORKER_BOSS_OPTIONS`) — it never sets `supervise: true`. `apps/api/src/server.ts`
passes no override either. Result: **no process in this codebase ever starts
pg-boss's maintenance engine** (`Boss#onSupervise` — gated by `if (this.#config.supervise)`
in `pg-boss/dist/index.js:88`). That engine runs `failJobsByTimeout`
(`plans.js:1001`, `WHERE state='active' AND started_on + expire_seconds < now()`)
— the exact mechanism that should reap a stranded `active` job. Because it never
runs, a job that goes `active` and then the worker dies stays `active` forever,
and the `exclusive` singleton key permanently blocks re-enqueue. This matches
prod evidence exactly (`maintain_on`/`monitor_on` null, `active_count` stale at 0).

The route-level "surface drops instead of silent no-op" fix (issue proposed-fix #3)
is **already implemented** on this branch — `POST /api/connectors/google/sync`
already returns `{ enqueued, deduped, jobId }` distinctly
(`packages/connectors/src/routes.ts` ~line 165-179). No change needed there.
Verified by reading the current route handler — stale spec/issue premise, not
re-building it.

## Fix

Enable `supervise: true` on the worker (the long-running process; mirrors the
existing sole-owner pattern already used for `schedule: true`). This turns on
pg-boss's real, already-tested `failJobsByTimeout` reaper for every queue in the
system, not just connector sync — the correct "shared supervision path" fix per
the handoff's collision note ("do not patch only one route").

Fork approved by Coordinator: pg-boss supervision also needs worker-only internal
maintenance grants. Add a grant file under `infra/postgres/grants/` for the
pg-boss schema only; do not touch app schema, RLS policies, payloads, routes, UI,
or app-runtime privileges.

behavior per the handoff and it adds narrow pg-boss internal grants to the
worker role. Security check: confirm app runtime is unchanged, no RLS/app-table
grants were added, no payload key was added, and no route/UI behavior changed.
