## 1. Problem

When Jarvis answers a question, it increasingly does so using context it gathered before the model
ever saw the turn:

- #530 passively recalls memory-graph facts before context-dependent turns;
- #525 fans out read-only reads across notes, email, calendar, and tasks and injects a hidden
  `<cross_tool_context>` block;
- #528 seeds compact core memory at launch.

All of that context is **hidden**. The user sees a confident answer but has no idea *which* sources
produced it. That breaks three things:

- **Verify:** the user can't confirm the answer is grounded in real data vs. model confabulation.
- **Trust:** "the contractor bid is due Friday" reads identically whether it came from a real
  calendar event or a hallucination.
- **Follow up:** the user can't jump to the underlying note, task, or event to act on it.

Jarvis already *has* the provenance — the pre-submit planners (#530) and collectors (#525) know
exactly which sources were consulted and how many items each returned. Today that information is
discarded after the hidden context block is rendered. #539 surfaces it.

## 2. Decision

Add **per-turn source provenance** to chat answers.

For every assistant turn whose answer was informed by pre-submit context retrieval, attach a
**metadata-only** provenance manifest describing *which source types were consulted and how many
items each contributed* — never the source content itself. The web client renders a compact
"Sources" affordance below the answer, with a deep link to the underlying source where one exists.

Provenance is **coordinator-assembled from the pre-submit retrieval/collection results**, not
generated by the AI model. This is the load-bearing trust decision (see §5).

V1 is intentionally narrow:

- metadata only (source type + count + optional deep link + optional timestamp);
- assembled from #530 and #525 pre-submit results plus #528 launch core-memory seed;
- one provenance manifest per assistant turn, owner-scoped, persisted with the turn;
- no inline per-sentence citations, no streaming, no AI-authored attributions.

## 3. Current Architecture Anchor

The relevant existing seams:

- `ChatSessionManager.runTurn()` owns the per-turn path: ensure session, submit text, drain the
  reply, persist. #530 and #525 add pre-submit retrieval/collection steps *inside* `runTurn()`
  before `session.engine.submit(...)`.
- `DataContextChatPersistence.recordTurn()` records the completed turn (user text + assistant reply)
  under `DataContextDb` via `ChatRepository.recordCompletedTurn()`, returning the persisted
  assistant message.
- `packages/shared/src/chat-api.ts` defines `ChatMessageDto` (the wire shape for a stored message)
  and its JSON Schema. The assistant message is the "turn output" surfaced to the client.
- #530 `planPassiveRetrieval()` + `MemoryRecallService.recall()` produce ranked memory items with
  `kind`, `confidence`, `provenance`, `validFrom/validTo`, and `sources`.
- #525 normalizes every tool result into `CrossToolEvidenceItem { source, title, summary,
  sourceLabel, occurredAt?, startsAt?, dueAt?, relevance }`.

**Naming reconciliation.** The issue text references `ChatTurnDto`. The codebase has no `ChatTurnDto`
— the stored-turn wire type is `ChatMessageDto`. This spec adds the new field to the **assistant**
`ChatMessageDto` (the turn output). Wherever the issue says "ChatTurnDto.sourceProvenance," read it
as "the `sourceProvenance` field on the assistant `ChatMessageDto`." No new top-level DTO is
introduced; introducing one would be a larger contract churn than the feature warrants.

#539 must reuse these seams. It must **not** import calendar/email/notes/tasks/memory repositories
directly from chat — provenance is built from the results the existing pre-submit steps already
return.

## 4. What Provenance Includes / Never Includes

### 4.1 Includes (metadata only)

A provenance manifest is an array of items, one per source type that contributed to the turn:

- **source type** — a closed enum label: `memory` | `notes` | `calendar` | `email` | `tasks`.
- **count** — how many items of that source type were consulted (e.g. `3` memory facts, `2`
  calendar events).
- **deep link** (optional) — a backend-derived in-app link to the source, when one exists for that
  source type (see §7). Absent for email in V1.
- **data timestamp** (optional) — the most relevant source-provided timestamp already present in the
  retrieval result (e.g. newest memory `last_confirmed_at`, soonest calendar `startsAt`, most recent
  email `occurredAt`, note modified time). Used to show "as of" freshness. UTC ISO 8601.
- **label** (optional, derived) — a short human display string the client can show, e.g.
  `"3 memory facts"`, `"2 calendar events"`, `"vault note: meeting-notes-06-27"`. When a single
  concrete item dominates (one note, one event), the label may name it using the **already-bounded
  source label** the collector produced (#525 `sourceLabel`); otherwise it is a type+count summary.

### 4.2 Never includes

- the source content itself (fact text, note body, email body/subject, event description, task
  notes);
- AI prompt text, persona, or any system instruction;
- raw retrieved text or the rendered `<retrieved_context>` / `<cross_tool_context>` blocks;
- connector payloads, credentials, auth tokens, session tokens, or any secret;
- raw private object ids (memory fact ids, note paths beyond the display label, message ids, event
  ids) **except** as opaque inputs to a backend-resolved deep link (§7) — the wire payload carries a
  resolved link, not the raw id where avoidable.

Provenance is a *receipt*, not a *transcript*. If a field would let a reader reconstruct private
content, it does not belong in provenance.

## 5. Trust Boundary (load-bearing)

Provenance metadata is **assembled by the coordinator (chat backend) from the pre-submit
retrieval/collection results, before and independent of the AI prompt.** The AI model never
produces, edits, or sees the provenance manifest.

Rationale: an AI-generated "Sources" list is prompt-injection-vulnerable and confabulation-prone. A
malicious note or email body injected into the hidden context block could instruct the model to
fabricate a citation ("Source: trusted-admin-policy"), or the model could invent a source that was
never read. Source-grounding the receipt to the *actual reads Jarvis performed* is the entire point;
deriving it from model output would defeat it.

Concretely:

- `runTurn()` collects the #530 memory recall result and the #525 evidence items it already computes
  pre-submit.
- A new pure `buildSourceProvenance(...)` step (see §6) folds those results into a metadata-only
  manifest **before** `session.engine.submit(...)`.
- The manifest is held aside during the reply drain, then persisted with the assistant turn in
  `recordTurn()`.
- The model's reply text is **not** parsed for citations. Nothing the model emits can add, remove, or
  alter a provenance item.

If a turn ran no pre-submit retrieval (e.g. a greeting #530/#525 both skipped), the manifest is empty
and no "Sources" section renders.

## 6. Data Model

### 6.1 Provenance item type

Add to `packages/shared/src/chat-api.ts`:

```ts
export type ChatSourceType = "memory" | "notes" | "calendar" | "email" | "tasks";

export interface ChatSourceProvenanceItemDto {
  readonly sourceType: ChatSourceType;
  /** Number of items of this source type consulted for the turn. >= 1. */
  readonly count: number;
  /** Backend-derived in-app deep link to the source, when one exists (§7). */
  readonly deepLink?: string;
  /** Most relevant source-provided timestamp, UTC ISO 8601. Freshness only. */
  readonly dataTimestamp?: string;
  /** Short display string, e.g. "3 memory facts" or "vault note: meeting-notes-06-27". */
  readonly label?: string;
}
```

### 6.2 Field on the assistant turn

Add an optional field to `ChatMessageDto`:

```ts
export interface ChatMessageDto {
  // ...existing fields...
  /**
   * Metadata-only provenance for sources consulted before this turn's answer.
   * Present (possibly empty) only on assistant messages whose answer ran pre-submit
   * retrieval. Coordinator-assembled, never AI-generated. Never carries source content.
   */
  readonly sourceProvenance?: readonly ChatSourceProvenanceItemDto[];
}
```

JSON Schema (`chatMessageSchema`): add `sourceProvenance` to `properties` as an array of a new
`chatSourceProvenanceItemSchema`, `additionalProperties: false`, with `sourceType` enum
`["memory","notes","calendar","email","tasks"]`, `count` as integer `minimum: 1`, and optional
`deepLink` / `dataTimestamp` / `label` strings. Keep `sourceProvenance` **out of the `required`
array** so existing/legacy messages and user messages remain schema-valid without it.

### 6.3 Persistence

The manifest is stored alongside the assistant message so it survives reload and history listing.
Add a nullable `source_provenance jsonb` column to the chat messages table via a **new** migration in
the chat module's `sql/` directory (never edit an applied migration; never place module SQL in
`infra/`). `ChatRepository.recordCompletedTurn()` accepts an optional provenance argument and writes
it on the assistant row; `listMessages()` maps it back onto `ChatMessageDto.sourceProvenance`.

Because the column is metadata-only JSONB and owner-scoped via the existing chat-message RLS, no new
policy is required (see §8). The repository must validate the manifest shape on write (closed
`sourceType` enum, `count >= 1`, no extra keys) so a malformed manifest can never persist.

### 6.4 Builder

Create `packages/chat/src/live/source-provenance.ts`:

```ts
function buildSourceProvenance(input: {
  readonly memory?: { itemCount: number; newestTimestampIso?: string; deepLink?: string };
  readonly crossTool?: readonly {
    readonly source: "notes" | "email" | "calendar" | "tasks";
    readonly sourceLabel: string;
    readonly timestampIso?: string;
    readonly deepLink?: string;
  }[];
}): readonly ChatSourceProvenanceItemDto[];
```

Rules:

- group `crossTool` evidence by `source`; emit one provenance item per source type with
  `count = items in that group`;
- emit a `memory` item only when #530 actually injected memory (item count > 0);
- `dataTimestamp` = the most relevant timestamp in the group (soonest future event for calendar,
  most recent for memory/email/notes/tasks); omit when none is available;
- `deepLink` per §7; for a multi-item group, link to the source's list/dashboard surface filtered to
  the consulted scope when such a surface exists, else omit;
- `label` = a type+count summary, or the single concrete `sourceLabel` when the group has exactly one
  item and that item has a safe display label;
- the builder is **pure and unit-tested**; it performs no I/O and reads no database.

The builder must never copy a `title`, `summary`, `excerpt`, subject, or body field from any
evidence item into the manifest — only `source`, count, a sanitized display label, a timestamp, and a
resolved link.

## 7. Deep Links

Deep links are **backend-derived** and point at existing in-app surfaces. This matches the
"No Note Viewer" rule — chat shows provenance plus a backend-derived deep link to the source, not an
inline viewer.

| Source type | Deep link target                              | V1 behavior                                            |
| ----------- | --------------------------------------------- | ------------------------------------------------------ |
| `notes`     | existing vault note deep link (note viewer/source open) | link to the consulted note; multi-note group → omit or link to notes search scope |
| `tasks`     | task detail surface                           | link to the consulted task; multi-task group → tasks view |
| `calendar`  | calendar event detail                         | link to the consulted event; multi-event group → calendar day/window |
| `memory`    | memory dashboard item (#533)                  | link to the memory dashboard, deep-linked to the fact when #533 ships; until then, link to the dashboard |
| `email`     | **none in V1**                                | label + count only; no deep link                       |

Constraints:

- the deep link is an in-app route string the web client already knows how to resolve; it carries no
  secret and no cross-user identifier;
- when a source group has more than one item and no clean aggregate surface exists, omit `deepLink`
  rather than guessing;
- if the underlying surface for a source type is unavailable (module disabled, #533 not yet shipped),
  omit `deepLink` and keep label + count;
- deep-link resolution happens in the chat backend at manifest-build/serialize time using
  already-available source identifiers; chat does not import the target module's internals — it
  builds a route string from a shared route contract (`packages/shared/*`).

## 8. Privacy, Security, and Invariants

- **Owner-scoped, no admin bypass.** Provenance is part of the chat turn; the chat message row is
  already owner-scoped under FORCE RLS. Admins do not bypass RLS to read another user's provenance.
- **Private by default.** Provenance is never shared across users. No cross-user deep links.
- **DataContextDb only.** All reads/writes go through the existing `DataContextRunner`; the new
  column is written within the same `recordTurn()` data context. No raw Kysely handle.
- **AccessContext unchanged.** No new fields; `{ actorUserId, requestId }` only.
- **Secrets never escape.** The manifest is metadata-only by construction (§4.2). It must never reach
  logs, pg-boss payloads, exports, or AI prompts as content. The episodic-embed and extract-facts
  job payloads stay metadata-only and are unchanged by this feature.
- **Module isolation.** Chat builds provenance from #530/#525 results and a shared route contract; it
  does not query notes/email/calendar/tasks/memory tables directly.
- **Provider-agnostic.** Provenance is independent of which model answered; no provider/model is
  hardcoded.
- **Trust boundary (§5).** AI output is never a provenance source.
- **Export / deletion.** Data export and account deletion already include chat messages; the new
  JSONB column rides along automatically. Confirm the export serializer includes it and the delete
  path purges it.
- **Logs.** Provenance assembly logs metadata only: actor id, thread id, per-source counts, whether a
  manifest was produced. Never log labels derived from private content, deep-link targets with ids,
  or timestamps tied to a specific private item beyond aggregate counts.

## 9. Chat Flow

In `ChatSessionManager.runTurn()`, extending the #530/#525 pre-submit path:

1. ensure session;
2. run #530 passive memory retrieval and #525 cross-tool collection (as those specs define);
3. from their results, call `buildSourceProvenance(...)` → a metadata-only manifest (pure, no I/O);
4. render the hidden `<retrieved_context>` / `<cross_tool_context>` blocks (#530/#525) and submit one
   combined engine payload — **the manifest is not part of the engine submission**;
5. drain the reply through the existing loop;
6. on successful completion, pass the manifest to `recordTurn(...)`, which persists it on the
   assistant message under `DataContextDb`;
7. the assistant `ChatMessageDto` returned by `listMessages()` / the live reply path carries
   `sourceProvenance`.

Failure behavior:

- if provenance assembly throws, log metadata only and persist the turn **without** a manifest — a
  provenance failure must never block or fail the user's answer;
- if the turn is stopped or the watchdog trips (no reply persisted), no manifest is persisted either;
- an empty manifest is treated as "no sources" — persist nothing for the field (omit) so the client
  renders no "Sources" section.

The manifest never triggers a second model call and never changes the answer text.

## 10. Web Client (Sources affordance)

`apps/web` renders a compact, authored "Sources" section below an assistant answer when
`sourceProvenance` is present and non-empty.

- one row per provenance item: source-type label + count (e.g. "Memory · 3 facts",
  "Calendar · 2 events", "Notes · meeting-notes-06-27");
- `deepLink`, when present, makes the row an in-app link to the source surface; otherwise the row is
  static text;
- `dataTimestamp`, when present, shows a relative "as of" hint;
- use the existing authored design system (`jds-*`, tokens in `apps/web/src/styles/tokens.css`); no
  new raw colors; no AI-tell accent left-border on these rows;
- the section is collapsed/secondary by default so it does not compete with the answer.

This is a **functional default** for the dogfood pass; the visual treatment can be annotated and
refined later. No streaming — provenance appears with the completed turn.

## 11. Out of Scope

- per-sentence / inline citations within the answer text;
- real-time provenance streaming as the answer generates;
- sharing provenance across users or any multi-user attribution;
- AI-generated source attributions or model-authored citations;
- email deep links (label + count only in V1);
- confidence/staleness UX beyond a single "as of" timestamp (#532, #541);
- a deep-linkable per-fact memory route before #533 ships (link to the dashboard until then);
- provenance for answers that ran no pre-submit retrieval (single-tool model-chosen tool calls mid-
  turn are not covered in V1; only pre-submit #530/#525 context is attributed);
- editing or correcting provenance from the UI.

## 12. Acceptance Criteria

- [ ] Assistant turns informed by pre-submit retrieval carry a metadata-only `sourceProvenance`
      manifest on `ChatMessageDto`.
- [ ] The manifest reports source type + count per source, with optional deep link and timestamp.
- [ ] The manifest **never** contains source content, prompt text, raw retrieved text, email bodies,
      connector payloads, or secrets.
- [ ] Provenance is assembled from #530/#525 pre-submit results, not from the AI reply; nothing the
      model emits can add, remove, or alter a provenance item.
- [ ] Greetings/control turns (no pre-submit retrieval) produce no "Sources" section.
- [ ] Provenance is owner-scoped via existing chat-message RLS; integration tests prove user A cannot
      read user B's turn provenance.
- [ ] Deep links resolve to existing in-app surfaces (notes/tasks/calendar/memory); email has none;
      unavailable surfaces omit the link but keep label + count.
- [ ] A provenance-assembly failure does not block or alter the user's answer.
- [ ] The new JSONB column is added via a new chat-module migration; no applied migration is edited;
      `foundation.test.ts` migration list is updated and full `test:integration` passes.
- [ ] Data export includes the provenance column; account deletion purges it.
- [ ] The web client renders a compact authored "Sources" section using the existing design system.

## 13. Verification

Local gate:

```bash
pnpm lint
pnpm format:check
pnpm typecheck
pnpm test:chat
pnpm test:shared
pnpm test:web
pnpm verify:foundation
```

Targeted tests:

- `buildSourceProvenance` folds #530 memory + #525 evidence into correct per-source counts;
- the builder copies no `title`/`summary`/`excerpt`/subject/body into the manifest (content-leak
  guard);
- the builder emits no `memory` item when #530 injected nothing;
- a malformed manifest is rejected on write (closed enum, `count >= 1`, no extra keys);
- `recordTurn` persists the manifest on the assistant row and omits it when empty;
- `listMessages` round-trips `sourceProvenance` onto `ChatMessageDto`;
- a thrown provenance assembly still persists the turn and returns the reply;
- deep-link resolution omits the link for multi-item groups with no aggregate surface, and for
  email always;
- model reply text containing a fake "Source: ..." string does not change the manifest
  (trust-boundary test);
- integration: user A cannot read user B's turn provenance;
- export includes the column; delete purges it;
- `foundation.test.ts` asserts the new migration row.
