# #1110 — Jarvis knows every SCREEN + the SETTINGS behind it (the app map)

Status: design (needs-spec → spec). Coupled with #1109 (runtime snapshot). Both share the
contracts in §7 of the #1109 spec; keep §5 here identical to that.

North star: **Jarvis answers from the map (ground truth) or says "I don't know" — never
invents.** #1109 gives Jarvis the symptom (current screen + error code). This spec gives
it the ground truth to resolve that symptom into a named fix, honestly.

---

## 1. Problem

App knowledge lives nowhere Jarvis can read. Screens and settings are declared with no
plain-language description (`ModuleSettingsSurfaceManifest` = id/label/path/scope/… — no
description; `JarvisModuleManifest` has no `requiredModel`/`requiredCapability`). The
news add-source prerequisite is a **buried code closure**
(`module-registry/src/index.ts:1380` — `hasJsonModel` calls
`resolveModelForService("module.news",{capability:"json",tierHint:"economy"})`); nothing
_declares_ it. So when the feature dead-ends, Jarvis has no declared fact to resolve it to
— it guesses. That is the origin failure.

**Live evidence of the risk:** the design brief itself misquoted the news error string
(claimed "Adding sources is unavailable right now — check your AI model…"; the real copy
is a `ClosedWriteGate` requirement + "Set it up in Assistant settings", server returns
`{status:"unavailable"}`). Prose _about_ the app already drifted from the app. The map
must be structurally resistant to exactly this.

## 2. Design

### 2.1 Enriched declarations (required, build-enforced)

Add a **required** plain-language `description` to every user-facing surface declaration:

- `ModuleSettingsSurfaceManifest` (+ nav entries, routes as applicable): "what this is /
  what it needs," one line.
- Enforcement is **two gates**, because a TS-required field alone is satisfiable with `""`:
  1. **Type**: `description: string` required — compile fails if the key is absent.
  2. **Registration-time assertion** (fail-closed, at module load — mirror the existing
     `requiresServices` build/test assertion in the gateway): reject a module whose
     surface has an empty/whitespace/over-length description. This is the real gate and
     also covers **third-party modules** compiled outside our build, consistent with ADR
     0009 compatibility gating.

### 2.2 Feature → dependency declarations that are LOAD-BEARING (anti-drift core)

The news prerequisite must be **declared once and consumed**, not declared as prose beside
a separate code check. Declare:

```
requires: { service: "module.news", capability: "json", tier: "economy" }
```

and have the availability gate (`hasJsonModel`) **derive from that same declaration**
rather than hand-coding the tuple. Single source of truth → the advertised requirement and
the enforced requirement cannot diverge. The declaration also carries a stable
`remediationRef` and a deep-link target (`/settings?section=assistant`).

This is finding **F3 (Critical)** folded in: a declaration that merely _describes_ behavior
drifts; a declaration that _drives_ behavior cannot. Prose "what this is" can still drift
(bounded, PR-reviewable, co-located) — accepted residual (§6).

### 2.3 Build step harvests declarations into a queryable map

A build step collects all descriptions + dependency declarations + deep-links + error
entries into a queryable **map artifact** shipped in the image (data, not source tree —
satisfies Ben's "no source in image"). Includes a **build stamp** (version + build id) so
#1109 can report app version and the map can answer "what version am I on."

### 2.4 Map read tool (sliced, capped, visibility-filtered)

A `risk:"read"` gateway tool returns **slices** of the map (by screen id, setting id, error
code, or query), never the whole map. Reuses `sanitizeAssistantToolResult` (key allow-list)

- `renderAndCap` (16k cap). Runs under the actor's `withDataContext`.

**F5 (Important) — visibility filter.** The map is app-structure metadata (same for all
users, no user rows → no cross-user data leak). But it must not leak **admin-only** or
**unreleased** surfaces to non-admins. The slice returned is filtered by the requesting
user: admin sections only to `isInstanceAdmin`, feature-flagged/unshipped surfaces gated,
inactive modules excluded. Enforce in the tool, not in the prompt.

### 2.5 Capability log (narrative only, non-authoritative)

Agents append a dated entry per PR (build exit-criterion). Powers "what can I do" and
"what's new in the last X." **F9 (Minor):** this is hand-written prose and WILL drift;
label it **narrative**, queryable for "what's new," but it is **never** the authority for
"what does X do / how do I fix Y" — those must resolve from §2.2 declarations. The tool
must keep the two sources distinct so the model can't cite the changelog as behavior.

## 3. Anti-hallucination — what's structural vs aspirational (F4)

"Answer from the map or say I don't know" is **not** enforceable purely by having a data
file; an LLM is not bound by a file's existence. Make it **as structural as achievable**:

1. **Closed-world framing (missing today, required):** the base system prompt must **not**
   contain app facts Jarvis can confabulate from. App structure/behavior/error knowledge
   lives **only** behind the map tool. Remove any ambient app description from the prompt.
2. **Provenance contract (§7 / #1109 §7.2):** an answer about app behavior/errors must
   follow a successful map/snapshot tool call; ungrounded app answers are a detectable
   failure.
3. **Adversarial eval suite (exit criterion):** a #1000 UAT eval asks known-unanswerable
   and known-answerable questions and asserts (a) answerable → cites the map slice,
   (b) unanswerable → returns "I don't know," never a fabrication.
4. **Reward the unknown + close the loop:** when Jarvis says "I don't know," log it →
   surfaces undeclared gaps as a coverage backlog for authors. Turns honesty into map
   growth.
   Residual: still probabilistic. So product copy must not promise perfection — see §6.

## 4. "No dead-end errors" is dishonest as stated → "no _unexplained_ errors" (F1, Critical)

The invariant only holds for errors whose cause is a **declared prerequisite**. Classes
that still dead-end and would recur the origin failure if we pretend otherwise:

- **transient/infra**: 503 web-search down, DB error, pg-boss job failure, timeout
- **external outage**: LLM 400/429/500, upstream API down
- **validation**: 422 "topic not allowed" (resolvable by the user, but not via a setting)
- **permission/RLS** denials
- **bug**: unhandled 500 / stack trace
- **third-party module** errors that declared no dependency

Resolution — reframe and make it structural via the **error contract** (§5 / #1109 §7.1).
Every first-party error carries `{code, class, remediationRef?}`:

- `class:"prerequisite"` → Jarvis resolves `remediationRef` in the map → **names the fix**.
- every other class → Jarvis **classifies honestly** ("this is a transient/system error,
  not something you can change in settings — here's what it means; retry / contact your
  admin") and **must not fabricate a settings fix**.

Without the structured `class`, Jarvis cannot tell a fixable prerequisite from a transient
failure and will invent a fix — the exact origin bug. So the error contract is a **hard
dependency**, not a nice-to-have. First-party surfaces must adopt it; third-party modules
that don't get the honest generic path (user-accepted risk, per constraint 3).

## 5. Shared contracts (identical to #1109 §7)

### 5.1 Structured error contract

```
{ code: string,            // stable, e.g. "news.add_source.no_json_model"
  class: "prerequisite" | "transient" | "validation" | "permission" | "bug",
  remediationRef?: string  // key into this map when class==="prerequisite"
}
```

### 5.2 Provenance contract

Assistant answers about app structure/behavior/errors must be grounded in a preceding
successful map/snapshot tool call; ungrounded app answers are a detectable eval failure.

## 6. Drift — residual risk and why it's acceptable (F3 continued)

- **Dependency behavior**: structurally drift-proof (§2.2 — one declaration drives both).
- **Plain-language "what this is" prose**: can still drift. Mitigation: co-located in the
  manifest next to the code it describes, so a PR diff shows both together; description is
  required so it can't silently vanish. Residual = a stale sentence, bounded, human-caught
  in review. Acceptable given Ben rejected source-mount and version-gating.
- **Capability log**: freely drifts, non-authoritative by §2.5. Acceptable.
- **Honesty backstop**: closed-world + provenance + eval means when the map _is_ stale,
  Jarvis is more likely to say "I don't know" than to confidently mis-answer. The failure
  mode degrades toward silence, not invention — which is Ben's stated preference.

## 7. Scope (F7, F8 Minor)

Correctly **two specs**. The shared contracts (§5, error taxonomy + provenance) span both;
duplicated verbatim here and in #1109 rather than spun into a third spec — keep in sync.
The capability log is a sub-concern of #1110, not a third spec; scoped narrow (§2.5).

## 8. UAT exit criteria (#1000 harness, hard gate)

- News add-source dead-end → chat "how do I fix this" → names json/economy model binding +
  deep-links Assistant & AI; grounded in the declared `requires`, not prose.
- Adversarial eval: unanswerable app question → "I don't know" (assert no fabrication);
  answerable → cites map slice.
- Transient-error case (e.g. web-search 503) → Jarvis says "transient/system error, retry
  or contact admin" and does **not** invent a settings fix.
- Non-admin map query does not return admin-only surfaces.
- Build fails when a settings surface ships with an empty description (assert the
  registration-time gate).

## 9. Explicitly out of scope

Live source reading in prod; source in image; source mount. Behavior knowledge = declared
tuples (§2.2), not the `.ts` tree.
