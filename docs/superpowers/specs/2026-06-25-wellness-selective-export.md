# Wellness: selective data export by timeframe + content (#484)

**Status:** approved
**Date:** 2026-06-25
**Owner:** Ben + Codex
**Grounded on:** `~/Jarv1s/packages/settings/src/data-export*.ts` (existing async export pipeline:
`enqueueExportBuildJob`, vault-stored at `exports/<jobId>`, `/api/me/export/download/:jobId`, auto-expiry

- cleanup #444), `packages/wellness/src/repository.ts` (`listCheckins` / `listMedications` /
  `listMedicationLogs` / `listTherapyNotes` / `insights`), Wellness-owned tables in
  `packages/wellness/sql/`.

## 1. Decision

A **Wellness-specific, selective export** — user picks a timeframe and which Wellness data categories
to include — generating a **clinician-friendly printable HTML document** (PDF via the browser's
print-to-PDF). Built as a **new export job kind owned by the Wellness module**, reusing the existing
async export pipeline (vault storage, status polling, download, auto-expiry) as infrastructure.

Distinct from the existing full-data JSON archive: selective (not all data), Wellness-scoped (not
all modules), and human-readable (a document for a therapist/doctor, not a dev JSON).

## 2. Entry point + modal UX

- **Entry:** an "Export" action on the **Wellness page** (per the issue). Opens a modal before
  generating anything.
- **Modal contents:**
  - **Timeframe:** a date-range picker (from / to). Default: last 90 days. Bounded by the user's
    earliest check-in and today.
  - **Content categories** (multi-select, all default-on):
    - Mood check-ins (core feeling, secondary, intensity, mood index, timestamp, note)
    - Medications (name, schedule) + medication logs (taken/skipped/prn, timestamp, "why I took it")
    - Therapy notes (body, linked check-in)
    - Insights (generated summaries covering the range, if any)
  - **Privacy confirmation** (required checkbox before Generate): _"This export will contain
    sensitive health and wellness data. Anyone you share it with will see it. Generate only if you
    trust the recipient (e.g. your doctor or therapist)."_
  - **Generate** button (disabled until timeframe valid + confirmation checked).

## 3. Format — printable HTML

The job renders a **self-contained, printable HTML document** (server-side string render, no
client-side data) and stores it in the vault at `exports/<jobId>.html`. Download returns the HTML
with a `Content-Disposition: attachment; filename="wellness-export-<range>.html"` header.

**Document structure:**

- Header: "Wellness export — \<owner name\>", date range, generated-at timestamp, a note that it was
  generated by Jarv1s.
- One section per selected category, each with a heading and the records (check-ins as a readable
  list/timeline; meds + logs as a schedule + log table; therapy notes as dated entries; insights as
  rendered summaries).
- Empty categories (no records in range) render a one-line "No \<category\> in this range" note —
  never silently omitted, so the recipient sees the category was considered.
- Footer: "Generated by Jarv1s on \<date\>. This document contains sensitive health information."

**PDF path:** the user opens the downloaded HTML in any browser → Print → "Save as PDF". No
server-side PDF pipeline (avoids headless-browser/pdf-lib deps in the container). The HTML is styled
for print (page breaks between sections, no app chrome). A one-line hint in the modal/download UI:
"Open in a browser and print to PDF to share."

**Redaction / omission:** the user controls what's included via the category multi-select; within a
selected category, all records in range are included (no per-record redaction in phase-1 — the
category toggle IS the redaction unit).

## 4. Generation — async job, reusing the export pipeline

New job kind `wellness-export` owned by `packages/wellness`:

- `POST /api/wellness/export` (Wellness route) body `{ from: date, to: date, categories: string[] }`
  → creates an export job row (reusing the `data_export_jobs` shape), enqueues a `wellness-export`
  pg-boss job with **metadata-only payload** `{ actorUserId, jobId, from, to, categories }` (dates +
  category names only — no health content in the payload, per CLAUDE.md invariant).
- The worker (registered by Wellness) loads the selected Wellness repos filtered by range, renders
  the HTML server-side (a pure template function in `packages/wellness/src/export-render.ts` — no
  React/DOM, just string templating with strict HTML-escaping of all values to prevent injection in
  the rendered document), writes to `exports/<jobId>.html` in the vault, marks the job complete.
- **Status + download + expiry reuse the existing `/api/me/export/*` routes.** The job row carries
  enough info (owner, format=html, filename) for the existing download handler to serve it; or
  Wellness adds a parallel `/api/wellness/export/download/:jobId` if cleaner module isolation wins.
  Decide in build; prefer reusing the existing download path if the archive-format field can carry
  "html".
- Job payload is metadata-only; the worker re-reads the timeframe + categories from the job row
  (DB), never trusts the payload for the data query (defense-in-depth against payload tampering).

## 5. Security, privacy & invariants

- **Owner-scoped throughout.** Export job, vault file, and all Wellness reads are owner-RLS-scoped.
  No admin bypass; an admin cannot generate a user's Wellness export (admin = config power only).
- **Sensitive-data acknowledgment gate.** The modal's required checkbox ensures the user explicitly
  consents to sensitive data leaving the app before generation. This is a UX/consent gate, not just
  copy.
- **Metadata-only job payload.** pg-boss payload = `{ actorUserId, jobId, from, to, categories }` —
  no check-in content, med names, or note text in the job payload (CLAUDE.md invariant). The worker
  reads content from the DB under owner scope.
- **HTML escaping is mandatory.** All user-generated content (notes, therapy-note bodies, med names)
  is HTML-escaped in the render template. A static test asserts no raw interpolation of repo values
  into the HTML (mirrors the briefing trust-boundary static check).
- **Auto-expiry + cleanup.** Reuses the existing export-expiry sweep (#444): the HTML file is deleted
  on expiry, same as JSON archives. Not stored indefinitely.
- **Audit.** Export generation + download are audited (`action: "wellness.export.generate" /
"wellness.export.download"`, metadata = `{ jobId, from, to, categories }` — no content).

## 6. Acceptance criteria (from #484)

- [ ] A user can start a Wellness export from the Wellness page (modal entry).
- [ ] The modal requires a timeframe + content-category selection + a sensitive-data confirmation
      before Generate is enabled.
- [ ] The generated export includes only the selected categories in the selected timeframe.
- [ ] The export is a clinician-friendly printable HTML document; opening in a browser + print → PDF
      is the share path.
- [ ] Empty categories render an explicit "no records" note (never silently omitted).
- [ ] The flow clearly signals sensitive Wellness data may be shared outside Jarv1s (modal copy +
      document header/footer).
- [ ] Async generation reuses the existing pipeline (status polling, vault storage, auto-expiry,
      download); job payload is metadata-only.
- [ ] Owner-scoped only; no admin bypass; no secrets in audit; HTML-escaped throughout.

## 7. Rollout / blast radius

- `packages/wellness/src/export-render.ts` — new (pure HTML template fn, strict escaping).
- `packages/wellness/src/export-routes.ts` — new `POST /api/wellness/export` (+ download if not
  reusing settings path).
- `packages/wellness/src/export-job.ts` — new worker for the `wellness-export` job kind.
- `packages/wellness/src/manifest.ts` — register routes + job.
- `packages/wellness/sql/` — possibly a tiny migration if the export-jobs table needs a
  `format`/`module` column to carry HTML + Wellness ownership; reuse the existing
  `data_export_jobs` table otherwise (no migration).
- `packages/shared/src/wellness-api.ts` — request/response DTOs + schemas.
- `apps/web/src/wellness/` — export modal component on the Wellness page.
- `apps/web/src/api/` — client fn + query key.

Reuses (no duplication): `data-export-jobs.ts` enqueue/expire machinery, vault write/read, the
existing status/download routes where possible.

## 8. Out of scope

- **Per-record redaction** within a selected category (category toggle is the redaction unit).
- **Server-side PDF rendering** (browser print-to-PDF is the path).
- **Direct recipient delivery** (email-to-therapist, fax, portal upload) — download + manual share
  only; delivery is out of scope and privacy-risky.
- **Re-import / round-trip** — export is one-way (share out), not a backup-restore format.
- **Other modules' selective exports** — Wellness is the first; the pattern (module-owned selective
  export reusing the pipeline) generalizes, but each module ships its own.
- **Password-protected / encrypted HTML** — the vault stores it owner-scoped; once downloaded it's
  the user's to protect. Defer encryption-at-download.
