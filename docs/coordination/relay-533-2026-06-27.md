## Task 5 failures — analysis

### Failure 1: PATCH /api/memory/graph/facts/:id → 500

**Root cause (confirmed):** Response schema mismatch.

In `packages/shared/src/memory-dashboard-api.ts:189`, the `patchMemoryFactDashboardRouteSchema` response 200 expects:
```javascript
200: {
  type: "object",
  required: ["fact"],
  properties: { fact: { type: "object", additionalProperties: true } }
}
```

But `dashboard-routes.ts` sends:
```typescript
return reply.code(200).send({ patched: true });
```

Fastify's fast-json-stringify tries to serialize `{ patched: true }` using a schema that requires `fact`. This throws during serialization → 500.

**Fix:** In `packages/shared/src/memory-dashboard-api.ts`, change patchMemoryFactDashboardRouteSchema response 200 to:
```javascript
200: {
  type: "object",
  properties: { patched: { type: "boolean" } }
}
```

Also check `patchMemoryEntityDashboardRouteSchema` response 200 — same schema shape issue (expects `{ entity: {...} }` but route sends `{ patched: true }`).

Similarly `postMemoryCandidateAcceptRouteSchema` response 200 expects `{ factId, entityId }` but route sends `{ accepted: true }`. Change to `{ properties: { accepted: { type: "boolean" } } }`.

### Failure 2: suppress/accept — status stays "pending" after route returns 204/200

**What's known:**
- reject test PASSES (same `#mark` path, same pattern)
- suppress/accept tests: route returns 204/200, but DB `getById` still shows "pending"
- `withDataContext` uses `rootDb.transaction().execute(...)` — proper Kysely transaction

**Hypotheses (none confirmed):**

1. **Silent transaction rollback.** `withDataContext` commits via Kysely transaction. If something after the UPDATE (before commit) throws and is silently swallowed, the transaction rolls back but Fastify somehow still sends the response. Unlikely but investigate by checking if suppress/accept services throw between `#mark` and callback return.

2. **Accept flow exception before markPromoted.** In `acceptCandidate`, `recallSvc.remember(...)` runs before `markPromoted`. If `remember` throws, `markPromoted` is never called. But then `acceptCandidate` would throw → route would 500, not 200. Unless `handleDashboardRouteError` is catching + sending 200? (Unlikely, it re-throws unknown errors.)

3. **insertPendingCandidate ON CONFLICT returning stale row.** The `ON CONFLICT` clause returns the EXISTING row if there's a signature match. Check if `sig + randomUUID()` is generating stable values (maybe test isolation issue). Add a `console.log(candidate.id, candidate.status)` in the test seed to confirm the returned candidate is actually "pending".

4. **Connection pool transaction isolation.** The pool has `maxConnections: 2`. If a connection's transaction state is not properly reset between calls... unlikely with Kysely.

**Recommended diagnostic:** Add `console.error` inside `#mark` after the execute to log the result:
```typescript
console.error('[DEBUG #mark]', status, id, result.rows.length);
```

Also add to the test for suppress:
```typescript
console.error('[DEBUG suppress]', res.statusCode, res.body);
```

This will show if the route is actually returning 204 and `#mark` sees >0 rows.
