#### [HIGH] Worker runtime retains UPDATE on chat_messages for a code path that no longer exists
**File:** `packages/chat/sql/0036_chat_worker_runtime_grants.sql:17` (also `packages/chat/sql/0035_chat_messages_update_grant.sql:5`)  
**Invariant violated / concern:** Least-privilege / private-by-default; dead grant widening attack surface. The repository header (`packages/chat/src/repository.ts:18-23`) states the legacy worker-backed `create/append/status/activity/complete` flow and pg-boss enqueue were removed in the "retire-legacy-chat-model" change, and the only writer left is `recordCompletedTurn`, which inserts messages born `stored` (`repository.ts:140-176`) and never updates them.  
**Detail:** Migrations 0035 and 0036 grant `UPDATE ON app.chat_messages` to `jarvis_app_runtime` and `jarvis_worker_runtime`, and 0036 creates the `chat_messages_update` RLS policy for both roles, expressly to support the `pending → working → stored | error` transition driven by the old worker (see the 0036 header). No code in `packages/chat/src` issues an UPDATE against `chat_messages` anymore (grep confirms: only `recordCompletedTurn` and `touchThread`/threads update). The worker also keeps `SELECT ON app.chat_threads` / `SELECT, UPDATE ON app.chat_messages`. These standing write grants on a table holding private conversation bodies are now pure attack surface with no live caller — exactly the kind of role power the "least-privilege roles, not conventions" memory warns against.  
**Suggested fix:** Add a new migration that `REVOKE`s `UPDATE ON app.chat_messages` from both runtime roles and `SELECT/UPDATE` from `jarvis_worker_runtime` where the in-process live runtime no longer needs them, and drop the now-unused `chat_messages_update` policy. Confirm the embed/extract-facts worker only needs SELECT (it reads via `listMessages`) and grant exactly that. Never edit the applied migrations — add a forward migration.

#### [HIGH] `incognito` is documented as immutable but nothing enforces it; flag can be flipped on an existing thread
**File:** `packages/chat/sql/0042_chat_memory_settings.sql:33-35`  
**Invariant violated / concern:** Private-by-default / data-integrity invariant asserted in a comment but not enforced. The migration says "Add incognito flag to chat_threads. Immutable once set; default false. Incognito threads are never embedded and never recalled into."  
**Detail:** The column is added with no trigger or `CHECK` enforcing immutability. The identity-change trigger `app.prevent_chat_thread_identity_change` (`0014_chat_module.sql:53-72`) guards only `id`, `owner_user_id`, and `created_at`; `enforce_chat_thread_update_scope` does not touch `incognito`. `chat_threads` has `GRANT ... UPDATE` to `jarvis_app_runtime` with an owner-scoped UPDATE policy, so the owner (or any code path under their context) can flip `incognito` from `false → true` or `true → false` on an existing thread. The privacy guarantee ("incognito threads are never embedded") is enforced only at `recordTurn` time by reading `thread.incognito` (`persistence.ts:104`); a thread toggled to non-incognito after the fact would then be embedded into long-term memory, defeating the stated guarantee, and an embedded thread later toggled to incognito leaves prior chunks in `memory_chunks`.  
**Suggested fix:** Either enforce immutability in `prevent_chat_thread_identity_change` (raise if `NEW.incognito <> OLD.incognito`) and document the one-way consequence, or delete the "Immutable once set" claim and design the toggle's memory side-effects explicitly. Do not leave a security-relevant invariant asserted only in a comment.

#### [MED] `extract-facts` queue is fully wired but the handler is a permanent no-op stub
**File:** `packages/chat/src/jobs.ts:104-111` (enqueued at `packages/chat/src/live/persistence.ts:111-116`)  
**Invariant violated / concern:** Dead-code / incidental complexity; spec-before-build scaffolding kept live. `handleExtractFactsJob` is a `TODO(phase3-facts)` that does nothing.  
**Detail:** Every non-incognito completed turn enqueues a `chat.extract-facts` job (`persistence.ts:116`) that runs, does nothing, and is reaped. The queue definition, payload type (`ExtractFactsJobPayload`), worker registration (`jobs.ts:145-153`), and per-turn `boss.send` all exist to feed a no-op. This is a standing line of empty jobs on every turn plus four supporting declarations for behavior that does not exist — the "remove dead vocabulary/scaffolding in the same pass" smell. It also wastes pg-boss throughput and obscures which jobs are real.  
**Suggested fix:** Until the capability-router fact extraction lands, do not enqueue `CHAT_EXTRACT_FACTS_QUEUE` from `recordTurn` and do not register its worker; keep the queue name constant only if a near-term spec is approved. Re-introduce the enqueue when `handleExtractFactsJob` actually extracts facts. Code-judo: this deletes a payload type, a queue def entry, a worker registration, and a per-turn send.

#### [MED] Codex MCP bearer token is interpolated into a tmux send-keys shell line (token in process args / ps output)
**File:** `packages/chat/src/live/cli-chat-engine.ts:236-256`  
**Invariant violated / concern:** "Secrets never escape … logs" / least-exposure. The session MCP token is a credential that grants tool execution as the user.  
**Detail:** For the `openai-compatible` (Codex) provider the engine builds `${tokenEnvVar}=${opts.mcpToken} codex ...` and sends it via `tmux send-keys` (`buildCodexCommand` returns the line; `submit`/`launch` send it). The token therefore appears in the assembled shell command, in the tmux pane scrollback, and — because it is an inline env assignment on the command — in the process's `ps`/`/proc` environment visible to other processes on the host. The code comments acknowledge this as an "accepted tradeoff for a local single-user session," but the project is explicitly multi-user (Phase 1 epics) and runs on a shared/LAN-reachable headless box. The Claude path injects the token via `--mcp-config` JSON on the command line too (`cli-chat-engine.ts:210-222`), and Gemini injects it via a settings file header (`launch:78-95`) — the file path is the only one that keeps the token out of argv.  
**Suggested fix:** Prefer file-based or stdin injection for all providers where the CLI supports it (Gemini already does). For Claude, write the `--mcp-config` JSON to a per-session 0600 temp file and pass `--mcp-config <path>` instead of inlining the JSON+token in argv. For Codex, set the env var on the tmux session environment (`tmux set-environment`) rather than as an inline argv assignment so it does not show in `ps`. At minimum, raise the severity of the comment's tradeoff to a tracked issue rather than an inline "accepted."

#### [MED] Persona file path is unguarded against `userName` injection into the CLI system prompt
**File:** `packages/chat/src/live/persona.ts:53-65` (rendered from `persistence.resolveUserName`, `persistence.ts:136-147`)  
**Invariant violated / concern:** Prompt-injection / trust boundary. `renderPersona` substitutes `{{userName}}` into the persona that becomes the CLI's `--append-system-prompt-file` content.  
**Detail:** `resolveUserName` returns `app.users.name` verbatim (only trimmed). That value is written into the persona context file and injected as system-prompt text into the live CLI (`cli-chat-engine.ts:228`). In a multi-user system where `users.name` is user-controlled (display name), a crafted name (e.g. containing fake `</system>`-style framing or instruction overrides) is injected into the assistant's system prompt for that user. This is self-injection (the user attacks their own session), so impact is bounded, but the persona is also the surface that disables tools and enforces the "no files/tools" stance — a user steering their own system prompt could undermine those instructions, and any future cross-user persona reuse would be a real escalation.  
**Suggested fix:** Treat `userName` as untrusted display data: strip control/markup characters and cap length before substitution in `renderPersona`, and keep the persona's security-relevant instructions (tool stance) in non-substituted text. Document that persona safety must not depend on the substituted token.

#### [MED] Turn timeout persists a fabricated assistant reply as a real stored message
**File:** `packages/chat/src/live/chat-session-manager.ts:243-258`  
**Invariant violated / concern:** Data-integrity / over-defensive internal turned into silent corruption. On poll-cap timeout the manager sets `reply = TIMEOUT_MESSAGE` and then records it as the assistant turn.  
**Detail:** When the engine never reports `complete` within `maxPolls`, the code emits an `error` record (good) but then calls `recordTurn(actorUserId, text, reply, …)` with `reply` = "Chat timed out before the model finished responding." (`chat-session-manager.ts:249-255`). That sentinel string is stored in `chat_messages` as a normal `stored` assistant message, gets embedded into long-term memory (non-incognito), and is replayed verbatim as "Assistant: Chat timed out…" into future sessions (`renderReplayBlock`, `listPriorTurns`). A transient timeout thereby poisons the persisted conversation and the recall corpus with a non-reply.  
**Suggested fix:** On timeout, either do not persist an assistant message at all (store only the user turn, or store with a distinct non-`stored` status that is excluded from replay and embedding), or mark it so `listPriorTurns`/the embed job skip it. Don't let an operational error string masquerade as model output in the durable record.

#### [MED] `getOrCreate` memory-settings upsert performs a no-op self-write under an UPDATE policy on every read
**File:** `packages/chat/src/memory-settings-repository.ts:24-39`  
**Invariant violated / concern:** Incidental complexity / over-engineered read. The "get" path is an `INSERT … ON CONFLICT DO UPDATE SET updated_at = updated_at`.  
**Detail:** Reading settings (`GET /api/chat/memory/settings`, and every `RecallService.recall`) issues an `INSERT … ON CONFLICT (user_id) DO UPDATE SET updated_at = app.chat_user_memory_settings.updated_at RETURNING *`. The self-assignment is a contortion to force `RETURNING *` on conflict, but it (a) requires the row to satisfy the UPDATE RLS policy / write grant on what is logically a read, (b) bumps the row's xmin/dead-tuples on every recall, and (c) is harder to reason about than a plain read-then-insert. Recall runs this on every session launch, so it is a write amplified across the hot path.  
**Suggested fix:** Replace with a `SELECT`; if missing, `INSERT … ON CONFLICT DO NOTHING RETURNING *` then re-`SELECT`, or use `INSERT … ON CONFLICT DO NOTHING` followed by a guaranteed read. Keeps the read path read-only and removes the self-write trick.

#### [LOW] `sanitizeInput` only strips a leading `!`; other CLI escape prefixes and the launch line itself are unsanitized
**File:** `packages/chat/src/live/cli-chat-engine.ts:276-278`  
**Invariant violated / concern:** Defense-in-depth completeness. `sanitizeInput` strips a leading `!` to block the bash escape hatch (matrix F4) but is applied only to user turn text.  
**Detail:** The comment scopes the risk to the `!` bash prefix, but interactive CLIs expose other prefixes (e.g. slash-commands like `/clear`, `/exit`, provider-specific `@`/`#` directives) that a pasted user turn could trigger and that the manager's offset/transcript model does not expect (a stray `/clear` mid-session would rotate the transcript away from the pinned path — the exact failure the `clear()` design avoids). Sanitization is a single regex with no test of the bracketed-paste boundary. The replay/seed blocks (`renderReplayBlock`, `renderMemorySeedBlock`) are submitted via the same `submit()` and are also only `!`-sanitized.  
**Suggested fix:** Document precisely which prefixes the bracketed-paste path already neutralizes (bracketed paste should make `/`-commands literal — verify in the spike) and, if any remain live, extend `sanitizeInput` or rely on bracketed-paste mode explicitly. Add a unit test asserting `/clear`-as-first-line does not rotate the transcript.

#### [LOW] Manifest `database.migrations` array is out of sync with the directory that is actually applied

**Invariant violated / concern:** Source-of-truth drift / misleading metadata. The runner applies all `.sql` in `migrationDirectories` by sorted filename (`packages/db/src/migrations/sql-runner.ts:100-123`), not the `migrations` array.

#### [LOW] Recall query embeds the actor UUID into the semantic query string
**File:** `packages/chat/src/recall-port.ts:72`  
**Invariant violated / concern:** Code-quality / correctness smell. `const query = \`${actorUserId} past conversations\`` embeds a UUID into the text that is embedded for vector search.  
**Detail:** A raw UUID prepended to the query text contributes noise tokens to the embedding with no semantic value (and is identical for all of a user's recalls), so it neither personalizes nor improves retrieval — RLS already scopes candidates to the actor. It reads as a leftover from an earlier design. Harmless but misleading and slightly degrades retrieval quality.  
**Suggested fix:** Use a static semantic query (e.g. `"past conversations"`) or, better, embed the user's current turn text for relevance. Drop the UUID from the embedded string.

#### [LOW] `injectRecord` is fully public and bypasses turn-at-a-time / persistence with no actor authentication of the record
**File:** `packages/chat/src/live/chat-session-manager.ts:318-320` (consumed by `gateway-notifier.ts:13-18`)  
**Invariant violated / concern:** Boundary clarity / least-API. `injectRecord(actorUserId, record)` emits directly into a user's SSE fan-out, trusting the caller's `chatSessionId === actorUserId` assumption.  
**Detail:** `ChatGatewayNotifier.emit(chatSessionId, record)` forwards straight to `manager.injectRecord(chatSessionId, …)` relying on the Phase 2 invariant "chatSessionId === actorUserId" (documented in the notifier). If that 1:1 mapping ever changes (multiple sessions per user, or session IDs decoupled from user IDs in a later phase), records would be fanned out to the wrong subscriber set with no guard. The method is exported on the manager's public surface with no narrowing.  
**Suggested fix:** Make the chatSessionId→actorUserId resolution explicit (a lookup the notifier performs) rather than an identity assumption, and/or document the invariant as a hard precondition with an assertion. Keep `injectRecord` internal to the gateway-notifier seam.
