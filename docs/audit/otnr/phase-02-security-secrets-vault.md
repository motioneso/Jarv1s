### Findings

#### [HIGH] Bearer-token auth path treats any opaque token as a session UUID — undocumented, untyped backdoor that bypasses better-auth
**File:** `packages/auth/src/index.ts:215-219`, `packages/db/src/auth-session.ts:11-31`  
**Invariant violated / concern:** Hard invariant #1/#2 (no bypass, private by default) + Error-handling/contract clarity. The "CLI-bridge auth primary / API key backup" path is a parallel authentication mechanism with weaker guarantees than the primary better-auth session flow.  
**Detail:** `resolveRequestAccessContext` short-circuits on *any* `Authorization: Bearer <x>` header and feeds `<x>` straight into `AuthSessionResolver.resolveAccessContext`, which runs `SELECT user_id FROM app.resolve_auth_session($1::uuid)`. This means the entire better-auth verification stack (CSRF/trusted-origin checks, cookie integrity, plugin hooks) is bypassed for any caller who presents a raw session id as a bearer token. The session id is the *only* secret — there is no separate API-key table, no per-key scoping, no revocation distinct from the session, and no rate-limit class on this path (rate limiting keys on IP, and the credential auth throttle in `apps/api/src/server.ts:157-163` only covers the `/api/auth/*` better-auth POSTs, not arbitrary bearer-authenticated module routes). A leaked session UUID (logs, referrer, browser history of the loopback OAuth redirect, etc.) is a full bearer credential usable headlessly. The `::uuid` cast does block SQL injection, and a malformed bearer throws "Invalid bearer token", so this is not a trivial break — but a session-id-as-API-key with no scoping/expiry-display/audit is a materially weaker credential than the cookie session it shadows, and it is effectively invisible (no type, no doc here, no audit event on use).  
**Suggested fix:** Make the bearer path explicit and least-privilege: introduce a dedicated, hashed, individually-revocable API-key table (store only a hash, never the token), scope keys to capabilities, emit an audit event on use, and gate the path behind an opt-in env flag. At minimum, document the invariant at the call site and confirm `resolve_auth_session` enforces `expires_at` server-side (verify migration 0046) so an expired session id cannot authenticate via bearer.

#### [MED] `resolveVaultPath` traversal guard relies on `resolve()` but does not reject absolute relative paths or symlink escape
**File:** `packages/vault/src/vault-path.ts:10-23`  
**Invariant violated / concern:** Hard invariant #3 (VaultContext for all vault I/O — the path guard is the enforcement point) + Security (path traversal / IDOR across user vaults).  
**Detail:** `resolveVaultPath(vaultRoot, relativePath)` calls `resolve(vaultRoot, relativePath)`. If `relativePath` is itself absolute (e.g. `/etc/passwd` or another user's `/data/vaults/<otherUser>/x`), `node:path.resolve` discards `vaultRoot` and returns the absolute path. The subsequent `startsWith(normalizedRoot + sep)` check *does* catch this (the resolved path won't be under the root), so traversal via `..` and via absolute paths is blocked — good. However: (a) the guard does not resolve symlinks (`realpath`), so a symlink created inside the vault (e.g. by a future module or a restored backup) pointing outside the root would pass the string check and be followed by `readFile`/`writeFile`; (b) all vault ops resolve against `ctx.vaultRoot` which is `mkdir`'d `0o700` but the traversal check is purely lexical. For a system holding personal data this is the single chokepoint and deserves defense-in-depth.  
**Suggested fix:** After the lexical check, on read/write resolve `realpath` of the parent dir (or the file) and re-assert it is within `realpath(vaultRoot)`; or document explicitly that vault contents are trusted and never symlinked, and add a test asserting absolute-path and symlink inputs are rejected.

#### [MED] Legacy (no-keyId) envelope decryption silently swallows every GCM auth-tag failure in a loop
**File:** `packages/connectors/src/crypto.ts:51-65`, `packages/ai/src/crypto.ts:51-65`  
**Invariant violated / concern:** Error handling (swallowed catches) + AES-256-GCM auth-tag verification clarity.  
**Detail:** For envelopes written before `keyId` existed, the code iterates `legacyCandidates` and `try { tryKey } catch {}` per key, treating any throw as "wrong key, try next". GCM authentication failure and a genuine bug (corrupt ciphertext, truncated base64, JSON parse) are indistinguishable here — every error becomes "try next key", and only after all keys fail do you get a generic "no key could authenticate it". This is acceptable for true key-rotation, but it (a) masks data-corruption from misconfiguration, and (b) `legacyCandidates` includes the dev-default-derived key when no env key is set, so in a misconfigured prod-like-but-NODE_ENV-unset deployment a legacy envelope could be silently decrypted with the well-known dev default. The two cipher files are also byte-for-byte duplicates (see LOW below).  
**Suggested fix:** Distinguish auth-tag failure (expected during rotation) from other errors — only treat the specific GCM auth error as "try next key"; rethrow JSON/decoding errors. Consider logging (without plaintext) when a legacy envelope is decrypted so operators can detect un-migrated data.

#### [MED] `resolveKeyring` derives AES keys via a single unsalted SHA-256 of the secret string
**File:** `packages/db/src/keyring.ts:33-51`  
**Invariant violated / concern:** Hard invariant #5 (connector/AI secrets AES-256-GCM encrypted at rest) — key-derivation strength.  
**Detail:** Both the current key and every retired key are derived as `createHash("sha256").update(secret).digest()`. SHA-256 of a passphrase is a fast, unsalted, single-iteration KDF. If `JARVIS_CONNECTOR_SECRET_KEY` / `JARVIS_AI_SECRET_KEY` are set to human-chosen passphrases (likely, given the env-var ergonomics) rather than 32 bytes of CSPRNG output, the derived key is brute-forceable far faster than a proper KDF would allow, weakening the at-rest guarantee for all connector OAuth tokens and AI API keys. The dev default `"jarv1s-development-..."` is a public string, so any non-prod deployment that forgets to set the env var encrypts real tokens under a globally-known key (only `NODE_ENV==="production"` forces the var — staging/LAN deployments are unprotected by default).  
**Suggested fix:** Require the key material to be 32 raw bytes (base64/hex) and validate length, or run the passphrase through a real KDF (scrypt/argon2) with a fixed deployment salt. Tighten the "required" check to fire outside `NODE_ENV==="production"` too (e.g. any non-`development`/`test` env), so LAN/staging cannot fall back to the public dev key.

#### [MED] `selectProviderWithCredential` returns the raw encrypted credential alongside the "safe" row shape via a double `as` cast
**File:** `packages/ai/src/repository.ts:283-311`  
**Invariant violated / concern:** Hard invariant #5 (secrets never escape) + TypeScript (cast-heavy contract obscuring the real invariant).  
**Detail:** This method selects `encrypted_credential` and casts the result to `AiProviderConfigSafeRow & { encrypted_credential }` with a trailing `as Promise<...>`. The "safe row" naming everywhere else in the file is the load-bearing convention that prevents secrets reaching serializers; this method deliberately punches through it, and the only thing keeping the credential from leaking is the doc-comment promise that callers "never log or forward" it. As of this audit `HttpApiAdapter` (the intended consumer, `packages/ai/src/adapters/http-api.ts`) is referenced only by its own module — no production caller wires `selectProviderWithCredential` → decrypt → adapter yet — so the method is currently an unguarded loaded gun with no compile-time barrier against a future caller serializing it.  
**Suggested fix:** Return a distinct branded type (e.g. `AiProviderWithSealedCredential`) whose `encrypted_credential` field cannot be passed to any serializer, and have the decrypt step consume-and-discard it. Drop the `as Promise<...>` cast by typing the Kysely select precisely. Until a real consumer exists, consider gating the method behind the M-A3 spec so it isn't dead surface area.

#### [MED] Google token-endpoint errors interpolate the raw upstream response body into the thrown Error
**File:** `packages/connectors/src/oauth.ts:109-112`  
**Invariant violated / concern:** Hard invariant #5 (secrets never escape via logs/errors) + Error handling (leaked internals in errors).  
**Detail:** `postToken` does `const detail = await response.text(); throw new Error(\`Google token endpoint returned ${response.status}: ${detail}\`)`. The `params` posted include `client_secret` and `refresh_token`. Google's OAuth error responses generally do not echo the secret back, but this is provider-controlled text that is then thrown up through `completeAuthorization`/`getFreshAccessToken`. In the connectors route `handleRouteError` (`packages/connectors/src/routes.ts:392-419`) a `GoogleConnectError` is sent verbatim, but a plain `Error` from `postToken` falls through to the final `throw error`, which Fastify will log (server `logger: true`) and, depending on the error serializer, may surface in the response. Coupling secret-bearing requests to verbatim upstream-body error messages is a latent leakage vector.  
**Suggested fix:** Do not include the upstream response body in the thrown message — log a sanitized status + a fixed code, and surface a generic "Google token exchange failed (status N)" to the user. If detail is needed for debugging, scrub it for any substring matching the submitted `client_secret`/`refresh_token` before logging.

#### [LOW] `ConnectorSecretCipher` and `AiSecretCipher` are byte-identical duplicate implementations
**File:** `packages/ai/src/crypto.ts:1-93`, `packages/connectors/src/crypto.ts:1-93`  
**Invariant violated / concern:** Code quality (duplicate helper instead of a single canonical utility) — the thermo-nuclear "delete complexity" mandate.  
**Detail:** The two files differ only in the type name (`EncryptedAiSecret` vs `EncryptedConnectorSecret`), the class name, the error string ("AI" vs "connector"), and the `createXSecretCipher` env-var arguments. The envelope shape, encrypt/decrypt logic, legacy-candidate loop, and validation are identical. Two copies of crypto code means a fix (e.g. the MED auth-tag-discrimination fix above) must be applied twice and can drift. Module isolation does not require duplicated crypto — both already depend on `@jarv1s/db`'s `resolveKeyring`/`Keyring`.  
**Suggested fix:** Hoist a single `JsonSecretCipher` (generic over a branded envelope type) into `@jarv1s/db` (alongside `resolveKeyring`) or a small shared crypto package; have both modules construct it with their own env-var names and brand. Deletes ~90 lines and a whole class of drift.  

#### [LOW] `encryptJson` uses a non-null assertion on the current key lookup
**File:** `packages/ai/src/crypto.ts:18`, `packages/connectors/src/crypto.ts:18`  
**Invariant violated / concern:** TypeScript (unjustified non-null assertion muddying the real contract).  
**Detail:** `const key = this.keyring.keys.get(this.keyring.currentKeyId)!;` asserts the current key is always present. This holds today because `resolveKeyring` always `keys.set(currentKeyId, currentKeyBuffer)` — but the invariant lives in another package, so the `!` hides a real cross-package coupling. If a future refactor of `resolveKeyring` changes that, this silently produces `undefined` passed to `createCipheriv`, which throws a confusing low-level error.  
**Suggested fix:** Replace the `!` with an explicit `if (!key) throw new Error("Keyring is missing its current key")`, making the invariant local and the failure mode legible.  

#### [LOW] `decryptJson` envelopes use `as` casts on JSON-parsed external data without runtime field validation
**File:** `packages/ai/src/repository.ts:195`, `packages/ai/src/repository.ts:245`, `packages/connectors/src/google-connection.ts:77-80`, `packages/connectors/src/google-connection.ts:115`  
**Invariant violated / concern:** TypeScript (unsafe cast of `unknown` to a precise shape) + Error handling (missing boundary validation).  
**Detail:** Decrypted payloads are cast straight to `GoogleConnectionSecret` / `{ clientId; clientSecret }` etc. via `as`, and DB JSONB columns are cast to `EncryptedConnectorSecret` (`row.encrypted_secret as EncryptedConnectorSecret`). `decryptJson` validates only that the payload is a non-array object — not that the expected fields exist or are strings. A malformed or partially-migrated row therefore yields `undefined` fields that flow into `oauthClient.refreshAccessToken({ refreshToken: undefined })` etc., producing opaque downstream failures rather than a clear "corrupt secret" error.  
**Suggested fix:** Add a narrow runtime guard (e.g. assert `clientId`/`clientSecret`/`refreshToken` are non-empty strings) immediately after `decryptJson` in `google-connection.ts`, and a guard when reading `encrypted_secret`/`encrypted_credential` JSONB before use.  

#### [LOW] `SessionTokenRegistry` / `ConfirmationRegistry` are unbounded in-memory maps with no eviction
**File:** `packages/ai/src/gateway/session-tokens.ts:19-47`, `packages/ai/src/gateway/confirmation-registry.ts:12-35`  
**Invariant violated / concern:** Code quality / robustness (resource leak) — not a secret-leak per se, but these registries hold session-scoped identity tokens.  
**Detail:** `SessionTokenRegistry.tokens` grows on every `mint` and only shrinks on explicit `revoke`/`revokeBySessionId`. If a chat engine is launched but never cleanly reaped (crash, dropped connection), its minted `jst_*` token — which maps directly to `{actorUserId, chatSessionId}` and is the *only* identity check in `AssistantToolGateway.callTool` — lingers in memory indefinitely and remains valid. There is no TTL. Over a long-lived API process this is both a slow memory leak and a pool of stale-but-valid identity tokens.  
**Suggested fix:** Give minted tokens a TTL and verify expiry in `verify()`; sweep expired entries. Tie token lifetime to the chat session's max lifetime so a leaked/orphaned token cannot authorize tool calls forever.  

#### [INFO] AES-256-GCM construction is correct: 96-bit random IV per encryption, auth tag stored and verified
**File:** `packages/ai/src/crypto.ts:17-31`, `packages/connectors/src/crypto.ts:17-31`  
**Invariant violated / concern:** Reviewed — clean (Hard invariant #5).  
**Detail:** Each `encryptJson` generates a fresh `randomBytes(12)` IV (correct 96-bit size for GCM), uses `aes-256-gcm`, stores the IV and `getAuthTag()` in the envelope, and decryption calls `setAuthTag` before `final()` so the tag is verified (a forged/modified ciphertext throws). IVs are never reused across encryptions of the same key (random per call; collision probability negligible at expected volumes). Envelope is versioned (`version: 1`) and algorithm-checked on decrypt. This is a correct authenticated-encryption-of-JSON design.  

#### [INFO] API serializers and the user-export script consistently project secret columns as `IS NOT NULL` booleans
**File:** `packages/ai/src/routes.ts:581-600`, `packages/connectors/src/routes.ts:286-301`, `scripts/export-user-data.ts:159-179`, `scripts/export-user-data.ts:291-370`  
**Invariant violated / concern:** Reviewed — clean (Hard invariant #5).  
**Detail:** `serializeProvider`/`serializeAccount` emit `hasCredential`/`hasSecret` (never the ciphertext); CLI-auth providers additionally force `hasCredential: false`. The export script projects `access_token IS NOT NULL AS hasAccessToken`, `password IS NOT NULL AS hasPassword`, `encrypted_secret IS NOT NULL AS hasSecret`, `encrypted_credential IS NOT NULL AS hasCredential` — never the raw secret/hash/token. `input_summary` for assistant actions stores only sorted key names + count (`summarizeAssistantToolInput`, `packages/ai/src/routes.ts:572-579`), not values. pg-boss chat jobs (`packages/chat/src/jobs.ts:26-32`) carry only `actorUserId`/`threadId`/`messageId` — metadata-only, satisfying Hard invariant #6; the worker re-fetches content under RLS. No `console.*` exists anywhere in the audited packages.  

#### [INFO] Vault I/O is correctly funneled through VaultContext; no raw `fs` on vault paths outside vault-ops
**File:** `packages/vault/src/vault-ops.ts:1-69`, `packages/vault/src/vault-context.ts:24-40`  
**Invariant violated / concern:** Reviewed — clean (Hard invariant #3).  
**Detail:** All read/write/list/delete operations require a branded `VaultContext` (obtained only via `VaultContextRunner.withVaultContext`, which derives the per-user root from `accessContext.actorUserId` and creates it `0o700`). `assertVaultContext` brand-checks at the boundary.
